01

Core Technology Stack

BayOne builds with the same tools it uses internally for its own quality engineering practice. The recommended stack is deliberately conservative: battle-tested frameworks that have reached production maturity, are extensible, and are maintainable by the team that will eventually own the system. Every component has crossed the threshold from experimental to production-grade.

Layer Technology Purpose
Browser automation Playwright Visual testing, UI interaction, screenshot capture, cross-device and cross-browser execution. Microsoft-native, aligned with Azure environments. Eliminates the delay and uncertainty issues that legacy Selenium-based automation introduces.
Agent orchestration LangGraph Multi-agent workflows, state management, tool routing, deterministic-to-agentic flow control. Manages the coordination between exploratory agents, review agents, and the playbook engine.
Model hosting Azure AI Foundry LLM inference and visual understanding models with enterprise-grade hosting, security, and scaling. Supports multi-model selection, allowing the platform to use the best model for each task type.
Workflow orchestration Apache Airflow Scheduling and orchestrating test suites at scale: nightly regression sweeps, post-deploy validation, periodic production monitoring, and batch processing of large test libraries.
CI/CD integration GitHub Actions Event-driven test triggering on code changes. Developers receive visual QE feedback within their existing workflow without context-switching to a separate testing tool.
Web application Django or FastAPI Unified platform hub providing user management, repository integration, SSO, playbook governance, configuration management, and reporting. Python-native for direct compatibility with LangGraph, Playwright, and Azure SDK, eliminating translation layers between the application and the agent infrastructure.

02

Design Principles

The architectural decisions behind the technology stack reflect a set of principles that govern how the platform is built and how it behaves.

Deterministic-First

Deterministic processes handle everything they can handle reliably. Playwright executes browser actions with precision. Saved playbooks replay exact sequences without interpretation. CI/CD triggers fire based on mapped code changes and recursive dependency analysis. AI is layered on top only for work that requires judgment: exploring unfamiliar application states, interpreting visual discrepancies, distinguishing intentional changes from defects, and generating new testing workflows.

This layering keeps the reliability floor high. The deterministic foundation operates at near-100% consistency. The AI layer adds capability without introducing variability into the critical path.

Multi-Model Strategy

Visual quality engineering benefits from selecting models based on what each does best rather than standardizing on a single provider. General reasoning and test orchestration use one model family. Visual understanding and pixel-level comparison may use another. This flexibility is managed within the platform through Azure AI Foundry and is transparent to end users. Model selection is a platform configuration choice, not a user-facing decision.

Recursive Dependency Coverage

Coverage extends beyond surface-level file associations. When the platform maps the codebase, it traces recursive downstream dependencies so that a change in one module triggers tests for everything that module affects, not just the module itself. This means the CI/CD integration targets the right tests on every change without executing the entire suite or relying on manual selection.

Automated Consistency at Scale

At scale, automated visual consistency exceeds what human reviewers can maintain. Across large volumes of pages, device and browser combinations, and A/B test variants, attention fatigue degrades human accuracy. The platform maintains consistent precision regardless of volume, and the deterministic playbook library ensures that validated tests execute identically every time.

Framework determines the ceiling: When platform performance plateaus, the constraint is typically in the framework and the deterministic foundation, not in the AI models. Investing in framework quality (the deterministic layer, the codebase mapping, the feedback loops) yields more than investing in better models. This insight informs every architectural decision in the platform.

03

Production Infrastructure

In production, the platform runs entirely on Azure, consistent with the existing cloud environment. The infrastructure follows a pay-by-usage model with no fixed commitments.

Component Service Characteristics
Application hosting Azure Container Apps Containers with compute allocated based on actual usage. Automatic scaling up during high-demand periods (nightly regression sweeps, post-deploy validation) and down during quiet periods. No idle infrastructure cost.
Database Azure PostgreSQL Default recommendation for test results, playbook metadata, configuration state, and confidence records. Flexible to accommodate alternative databases based on team preference, including on-premises options if required.
Model inference Azure AI Foundry Managed model hosting with enterprise security, compliance, and scaling. Supports multiple model deployments for the multi-model strategy.
Caching Azure Redis Available as needed for performance optimization, session management, job queuing, and real-time result streaming to the platform hub.
Development environments Docker Isolated, reproducible execution environments for development and testing. Every test runs in a clean, known state. Docker containers mirror the production Azure Container Apps configuration.

Cost Model

The infrastructure follows a pay-by-usage model. Azure Container Apps allocate compute based on actual demand and scale automatically, ensuring cost remains proportional to usage at all times. There is no fixed infrastructure commitment.

During early stages (single consumer group, initial playbook library), the platform runs on minimal compute. As adoption grows across consumer groups and the playbook library expands, the infrastructure scales smoothly without rearchitecting. This means the investment grows with the value delivered, not ahead of it.

Production-grade from day one: The development and production environments share the same architecture (Docker containers locally, Azure Container Apps in production). There is no prototype that requires rebuilding for scale. The platform that the team develops against is architecturally identical to the platform that runs in production.

04

Integration and Extensibility

The platform is designed to connect with existing tools and workflows without requiring changes to the systems it integrates with.

Model Context Protocol (MCP)

The platform's modular architecture natively supports MCP integration, consistent with the organizational requirement that all new tooling integrates seamlessly into existing workflows through MCP servers. This is a standard capability of the architecture, not an add-on. MCP connectors are available for Figma (design file references for development teams), Canva and WordPress (content and marketing tool integration), and additional connectors can be built as needed for any tool that supports the protocol.

Repository and CI/CD Integration

The platform connects to code repositories through the unified hub and through GitHub Actions for CI/CD triggers. When a developer pushes a change, the platform receives the event, identifies which visual tests are relevant based on the codebase map, and executes them. Results flow back to the developer within their existing workflow. No context-switching to a separate testing application is required.

Single Sign-On and Access Control

The platform hub integrates with existing SSO infrastructure. Each user sees the platform scoped to their role: QE teams see their test results and playbook library, development teams see CI/CD-triggered feedback on their changes, UI/UX teams see governance dashboards and approval workflows, content production teams see publishing validation reports, and leadership sees aggregate performance and coverage metrics.

Extensibility Beyond Visual QE

The modular architecture is not limited to visual quality engineering. The same agent orchestration layer (LangGraph), the same playbook engine, and the same observability framework can support additional testing types (API testing, accessibility compliance, performance monitoring) as organizational needs evolve. Each new capability is an addition to the platform, not a separate system. The shared knowledge generated by the platform (codebase maps, playbook libraries, traceability data, confidence metrics) can also complement broader organizational initiatives around unified knowledge bases and cross-functional quality standards.

Designed for ownership: Every technology choice in this stack is made with long-term ownership in mind. The platform uses standard, well-documented frameworks with large community support. The team that eventually owns this system will find extensive documentation, community resources, and readily available talent for every component. BayOne does not build platforms that create dependency on BayOne.
