# 03-04 Bridge: What Changed Between Andrew/Grishi Meeting and Technical Deep Dive

**Source:** Comparison of Sets 03 and 04
**Date:** 2026-03
**Type:** Bridge document

---

## What Was Validated

1. **Agents + MCP is the right approach.** Andrew restated the multi-agent orchestration vision. The entire Sephora technical team (Andrew, Grishi, Sergey, Malika) aligned with the agentic approach. No dissent.

2. **Cognos and DataStage are both accessible.** Colin's homework confirmed full Java/.NET SDKs for both Cognos and DataStage, going back to versions from 2003. The Set 03 feasibility question ("is it even possible?") is definitively answered: yes.

3. **The team wants to see it working, not hear about it.** Grishi said "We need to see a demo, really" early in the meeting. Sergey said "It's easy said than done." The demand for proof from Set 03 only intensified.

4. **Human-in-the-loop is expected.** Both sides agree: agents do the boring work, humans handle edge cases, business logic review, and final approval. No one expects full automation.

## What Was Corrected

1. **The demo does NOT require access to Sephora's environment.** In Set 03, the assumption was that connectivity to Cognos was a gating requirement. In Set 04, Malika proposed the disconnected approach: share exported XML and schema info, build the demo without touching Sephora's systems. This removed the security blocker entirely.

2. **Scope expanded: it's not just reports.** Grishi clarified that agents should also help with table data migration from SQL Server to Databricks, not just the report conversion side. Andrew confirmed: "I just want to make sure we cover both sides of the house."

## What Was New in Set 04

1. **Sergey entered the conversation.** He is the single most consequential new voice. His explanation of the existing AggregationApplication framework fundamentally shaped the demo:
   - No Scala code needed. Just YAML configuration files.
   - Three files per job: pipeline YAML, deployment YAML, create.HQL
   - Thousands of jobs already running on this framework in production
   - Existing support team already knows how to maintain it
   - Converting stored procedures to Spark SQL is simpler than writing new code
   - His ideal: "Ask the agent: we have 10 jobs, convert them, generate YAML files, commit to repo. Developer reviews and approves."

2. **Sergey's critique of Claude's output.** He described Claude producing "spaghetti code" Databricks notebooks with hundreds of windows that are impossible to debug or maintain. This is why the output MUST be YAML configs that fit the existing framework, not raw code.

3. **Malika's disconnected approach.** Share Cognos report XML + Databricks schema definition. Colin builds the demo without accessing Sephora's environment. Malika works with Vlad to get a Finance report.

4. **Two demo tracks crystallized:**
   - Track A: Cognos report XML conversion (lift-and-shift + remap to Databricks)
   - Track B: DataStage ETL conversion (stored procedures to Spark SQL + YAML configs)

5. **Security is a known blocker.** Sephora's security team previously blocked an MCP server attempt. This is accepted reality, not a surprise. Colin offered to help navigate IT.

6. **Colin's playbook concept.** Record a 30-45 minute call of someone walking through the manual process. That transcript becomes the agent's first input. This is how domain knowledge transfers to agents.

7. **Andrew Ho's framework model explanation.** Detailed walkthrough of how Cognos reports work: some point to framework models (SQL generated dynamically), others have free-hand SQL embedded. The second type requires manual SQL extraction and conversion. This is the exact process the demo automates.

8. **Sergey mentioned 85% success with Claude on DataStage conversion.** This is a higher baseline than Grishi's 30% from Set 03, but Sergey was specifically talking about DataStage job XML conversion, not Cognos report SQL.
