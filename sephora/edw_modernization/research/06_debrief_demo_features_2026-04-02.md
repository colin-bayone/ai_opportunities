### 2.5 Gate Results

Gate results are displayed as dropdown boxes -- one per gate. Each dropdown shows the gate name and its pass/fail score.

**Closed vs. open behavior (lines 463-464):**

- If a gate scores 100%, its dropdown box is displayed closed (collapsed). No action needed from the user.
- If a gate has issues (score below 100%), the box stays open and displays the specific flagged issues.

**Saurav (line 463):** "If the score is like 100% we will close the box. If it if we are having any issues here it will like keep it open and flag that issue exactly."

**Gate details per stage:**

- **Gate 1 (G1):** Checks if all source files and target examples are loaded, correct pipeline is selected, run ID is generated, artifact store is populated (lines 197-231)
- **Gate 2 (G2):** Validates parser outputs -- checks table names match source, column names exist in source files, transformations are present. Uses search-based matching to detect hallucinations. Cannot deterministically validate business logic (lines 294-312)
- **Gate 3 (G3):** Validates pattern mapping output (lines 360-361)
- **Gate 4 (G4):** Checks SQL generation -- verifies SQL is not truncated, all planned stages are present, logic is sound. Includes both deterministic checks (stage count matching) and an LLM check for business logic soundness. Uses a 75% accuracy threshold specifically for business logic (lines 411-436). This gate was added because previously business logic issues would slip through and cause wasteful token-burning retries at later stages.
- **Gate 5 (G5):** Validates configuration generation -- checks column names in deployment files, config parameters passed correctly (lines 456-457)

### 2.8 Auto-Fix Button

A button that performs surgical, targeted edits to the generated SQL based on the AI Review findings (lines 512-554).

**How it works:**

The system treats the generated SQL file, the source XML, Scala files, and all other artifacts as files in a workspace -- analogous to how Claude Code operates. It has tools to read and edit those files. When auto-fix is triggered, the AI:

1. Takes the review findings (bugs and advisories)
2. Opens the relevant generated SQL file
3. Goes to the specific line numbers where issues were found
4. Makes surgical edits to fix only those issues
5. After fixing, automatically re-runs validation
6. Returns to the review screen with updated results

**Saurav (line 547-553):** "You gave it everything in a file and you ask it then after the generation that go ahead and review this validation. As well as this generation. Then after it came up with the audit result, you set it to yeah, go ahead and fix it. Then it opens the SQL, go to that line number and fix those exact things. So kind of like surgical edits rather than like full generation."

**Implementation detail:** Each pipeline node operates as an individual chat thread (lines 594-597). The auto-fix leverages the existing conversation context from the review step -- it does not start fresh. The review findings, the validation results, the false positive flags, and the approval/rejection reasoning are all fed back into the same chat thread for the fix. This mirrors the experience of iterating with Claude Code in a single session.

**Saurav (line 594-596):** "So you can think of each of these nodes as like suppose individual chat threads. So it keeps in mind the previous context as well as previous generation."

**After auto-fix completes:** The system re-runs validation and returns to the review screen with new details. The user can then repeat the review-fix cycle or approve (lines 560-566).

### 2.9 Reject-and-Retry Button

An alternative to auto-fix that performs a full regeneration. When clicked, execution returns to the SQL Generator node and re-runs all downstream stages: SQL generation, Gate 4, config generation, config gate, validation, and then back to review (lines 518-525).

**Key difference from auto-fix:**

| | Auto-Fix | Reject-and-Retry |
|---|---|---|
| Approach | Surgical line edits to existing SQL | Full regeneration from SQL stage forward |
| Speed | Fast (single targeted edit pass) | Slow (re-runs 5+ pipeline nodes) |
| Token cost | Low | High (burns tokens on full regeneration) |
| DAG animation | Works correctly | Breaks -- DAG node lighting gets confused |
| Best for | Small issues (wrong table name, null handling, WHERE clause) | Fundamental structural problems |

**Saurav (line 525-526):** "I was just what you call kind of messing around with this. I found like the auto fix one, what you call more intuitive. Because we have already done like most of the work and then it comes down to like this final stage."

**Saurav (line 532-534):** "In total 25 or 26 transformations, it might be possible that there is one table name wrong or one where condition wrong or one null handling wrong... So these are kind of like small issues in total 26 transformation. So that's why I added this auto fix feature."

## 6. Execution Log Behavior: Detailed Technical Reference

The execution log behavior is important to understand so Colin can narrate it confidently during the demo without being surprised by visual pauses.

**Active spinner phase:** While an agent is processing, a spinner with a live-updating timer is visible. The text shows what the agent is doing (e.g., "parsing XML structure") and the elapsed seconds.

**Agent completion:** When an agent finishes, its timer stops. If the next agent starts immediately, a new spinner entry appears. If there is a gap (e.g., waiting for a gate), the spinner briefly disappears.

**Between-stage gaps (line 844):** After an agent completes and before the next stage begins, there may be a brief period with no visible spinner. This happens when gate checks are processing or when the system is waiting for parallel agents to all finish before proceeding. This is normal behavior, not a hang.

**Parallel agent visibility (line 892):** When multiple agents fire in parallel (e.g., the three parsers, or the three config generators), new spinner entries appear for each. Multiple spinners can be active simultaneously, visually demonstrating parallel execution.

**Throttling behavior (lines 851-866):** At concurrency of 4, the system fires up to 4 parallel API calls. If the API rate limit (75K tokens per minute for both Opus and Sonnet) is insufficient, 529 errors (throttling) occur. Saurav kept concurrency at 4 to stay within limits. Colin submitted quota increase requests (4-5x) for both Opus and Sonnet during the call. If the quota increase is approved, concurrency could be raised to 8-9 for faster parallel execution during the demo (lines 871-873).

## 7. AI Review Feature: What It Shows and Why It Matters

The AI Review is a post-validation audit that serves as a second-opinion layer. It was added specifically because the gate checks (particularly deterministic ones) produce false positives (lines 1424-1425).

**What gets sent to the AI:**
- All generated files (SQL, YAML, HQL, deployment config)
- All validation results from all gates
- All source files for cross-reference

**What comes back:**
- Findings categorized as bugs or advisories
- Each finding includes a descriptive explanation readable by a data engineer in one pass
- Visual highlighting draws attention to specific items requiring review
- An overall assessment (e.g., "recommended review" even when only advisories remain)

**Strategic value for the demo:** The AI Review transforms the system from "generates code and checks it" to "generates code, checks it, audits its own checks, and explains what it found in plain language." This is the feature that bridges the gap between the engineering pipeline and the business audience reviewing results.

## 8. Auto-Fix vs. Reject-and-Retry: Technical Comparison

### Auto-Fix (Use This for the Demo)

- **Mechanism:** Single AI agent with access to file read/edit tools makes targeted changes to the generated SQL at specific line numbers
- **Analogy:** "Claude Code for your migration pipeline" -- the AI reads the file, goes to the problem, and makes a surgical edit
- **Scope:** Fixes only the issues identified in the AI Review (e.g., wrong table name in one transformation out of 26, incorrect WHERE clause, null handling error)
- **After fix:** Automatically re-runs validation and returns to the review screen with updated results
- **Chat thread continuity:** Operates within the same chat thread as the review, so it has full context of what was found and why
- **DAG behavior:** Works correctly; DAG does not need to re-animate

### Reject-and-Retry (Do Not Use for the Demo)

- **Mechanism:** Returns execution to the SQL Generator node and re-runs all downstream nodes (SQL generation, Gate 4, config generation, config gate, validation, review)
- **Scope:** Full regeneration of all SQL -- not targeted
- **After retry:** Runs through 5+ pipeline nodes before returning to review
- **Token cost:** Burns significantly more tokens than auto-fix for typically small issues

## 10. Back-End Architecture Context for Demo Narration

While the demo focuses on the UI, Colin needs context on what is happening behind the scenes to narrate the pipeline stages confidently.

### Pipeline Stages in Order

1. **Orchestrator:** Loads source files, selects the correct pipeline based on file types, generates run ID, establishes artifact store (lines 199-231)

2. **Gate 1:** Validates all source artifacts loaded, target examples present, correct pipeline selected, run ID generated (lines 217-231)

3. **Three Parallel Parsers (lines 237-289):**
   - **Data Stage Parser:** Extracts job stages, transformations, source/target connections, data flow from XML. Uses deterministic extraction plus LLM synthesis in parallel (lines 255-262)
   - **SQL Interpreter:** Extracts stored procedures, table joins, CTEs, temp tables, business logic, aggregation patterns. Uses SQL Glot for deterministic parsing plus LLM (lines 268-276)
   - **DDL Parser:** Extracts table names, column names, data types, typecast info, constraints, precision from DDL and view files. Uses SQL Glot plus LLM (lines 276-289)

4. **Gate 2:** Cross-references parser outputs against source files to detect hallucinations and missing elements (lines 294-312)

5. **Pattern Mapper:** Creates a translation dictionary from SQL to Spark using all three parser outputs plus target Scala/YAML examples. Single Opus call because of the large context required (lines 320-358)

6. **Gate 3:** Validates pattern mapping output (line 360)

7. **SQL/Spark Generator:** Three-step process (lines 386-406):
   - Plans transformations and creates a dependency graph identifying independent vs. dependent stages
   - Generates all independent stages in parallel (max concurrency of 4)
   - Generates all dependent stages after independent ones complete

8. **Gate 4:** Deterministic checks (SQL truncation, stage count) plus LLM check for business logic soundness at 75% threshold (lines 411-436)

9. **Three Parallel Config Generators (lines 442-456):**
   - Pipeline YAML configuration
   - Deployment YAML configuration
   - Create HQL / HQL DDL

10. **Gate 5:** Validates configuration outputs (lines 456-462)

11. **Validation:** Basic deterministic validations (line 462)

12. **Review/Approval:** AI Review, Auto-Fix cycle, human approval (lines 484-570)

### Model Usage

- **Opus:** Used for heavy tasks requiring large context -- pattern mapping and SQL generation (lines 372-378)
- **Sonnet:** Used for lighter tasks (implied from the discussion of quota for both models)
- **Concurrency:** Set to 4 to stay within API rate limits. Could be increased to 8-9 if quota is approved (lines 851-873)

## 12. LangSmith as Back-End Observability (Not for Demo, But Available)

LangSmith is connected for tracing all pipeline runs. It provides a waterfall view showing every API call, node execution, and parallel processing pattern (lines 953-1009). Key capabilities:

- **Waterfall view:** Shows all stages with timing, parallelism visible as simultaneous bars
- **Node-level I/O:** Click any blue node (stage-level) to see inputs and outputs
- **LLM-level I/O:** Click any orange node (individual API call) to see exact prompts and responses
- **Structured state inspection:** Can drill into any LangGraph state field to see gate check results, parser outputs, generated SQL, etc.
- **Thread view:** Shows the full execution as a single thread with all turns

LangSmith is not real-time (updates within 1-2 minutes of execution, not instantly -- line 1023-1032). The execution log in the dashboard is the best real-time indicator.

**Saurav (line 1032):** "It does not like what you call update in real time, real time like the execution log itself is like the best place to see exactly what's happening."

**Saurav (line 1203):** "I was even like, do we even need a UI? Because Langsmith does a better job in my opinion."
