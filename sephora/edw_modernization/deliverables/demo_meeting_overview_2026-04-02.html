Key Takeaways

TAKEAWAY 01
The demo worked and impressed
The pipeline processed Sephora's actual data end-to-end in 13.5 minutes with no crashes or major issues. Andrew's closing was unambiguously positive. This is the first meeting in the engagement that ended without any new gate requested. No additional demos, no extra proof points, no "come back with more information." The evaluation phase is complete.
TAKEAWAY 02
Maher Burhan (the technical assessor) was satisfied
Maher asked 100 utterances worth of technical questions spanning validation, architecture, security, deployment, and error handling. Every question was answered. The Azure isolation confirmation killed the security objection in one exchange. His closing framing of the tool as "the transition period" is healthy and honest. He is not opposed. He is pragmatic.
TAKEAWAY 03
The exploration SOW pitch landed
Andrew asked "what would be step one?" and Colin delivered the two-week proposal exactly as planned: Week 1 guided discovery with SMEs, Week 2 autonomous complexity mapping, deliverable is a data-driven engagement proposal. Andrew responded with "Got it. OK." and moved to wrap-up. No pushback, no counter-proposal. He had actually started describing the exploration concept himself before Colin even pitched it.
TAKEAWAY 04
Five of seven new requirements need system access
Data reconciliation, migration detection, schema mismatch handling, live data reading for mapping, and Cognos MCP all require access to Sephora's production systems. This means the exploration SOW is not just strategically desirable, it is operationally necessary. These capabilities cannot be delivered without access. Access implies a contract.
TAKEAWAY 05
Portability and self-sufficiency is a real concern
Vishal Sharma explicitly asked about running the system without BayOne. Maher reinforced the "transition period" framing. This needs to be addressed clearly in the engagement proposal: the system is temporary, agents are customizable, Sephora keeps everything, and BayOne's goal is to make itself unnecessary for the migration tooling once the program is complete.
04

Key Questions Asked

29 technical questions were asked during the demo. Here are the most significant exchanges by topic.

Validation and Quality

"When you say validation, what are you validating against?"
Maher Burhan, [00:04:19]

Colin explained the two-tier validation: deterministic checks first (syntax, table presence, column matching via sqlglot), then LLM-based business logic verification. Deterministic gates must pass before LLM checks run. This saves tokens and ensures reliability.

"So which schema is it like the target schema or which schema validation is it doing?"
Grishi, [00:06:17]

Colin explained the orchestrator classifies all files and routes them to the correct parser. The Databricks schemas Sephora provided go to the DDL parser. The mapping between legacy and Databricks happens at the pattern mapper stage automatically.

Security and Architecture

"When you make calls to these models, do you go out of the our network to the public model or you have your private?"
Maher Burhan, [00:20:17]

Colin showed the architecture diagram and confirmed: "There are no calls to anything at this current moment in time that ever go outside of your Azure instance. Every single thing that we built here is entirely based on Azure exclusively." Maher responded: "So basically when you're finished with this, you're gonna tell us deploy it to your VNet, put it in your subnet and we'll be able to run it. It's not reaching out to anything." Colin confirmed.

Scaling and Parallelism

"It can be deployed to run in multiple in parallel in multiple business subject area in parallel. Am I correct? I just want to make sure that this is not like you still have to have a person feeding couple report at a time."
Andrew Ho, [00:50:13]

Colin confirmed: the only constraint is compute. FastAPI is lean, the heavy lifting happens in Azure AI Foundry. Adding Celery workers (~$20/month) or Redis (~$30/month) enables full parallel execution. He described the "orchestrator of orchestrators" pattern for running all business areas simultaneously.

Portability

"We want to make sure that we can run something fully in our environment without your system as well. So that you know, we're sure that in case we want to move ahead, we want to change it in the future."
Vishal Sharma, [00:35:03]

Colin confirmed agents are fully customizable, the system is temporary for the migration, and Sephora keeps everything. Maher reinforced: "This is the transition period. We use it to migrate. When we migrate, we don't need it anymore."

Next Steps

"In reality, if I was like day one, how are you gonna go about it?"
Andrew Ho, [00:45:55]

Colin proposed the two-week exploration SOW: guided discovery Week 1 (~8 hours of meetings with SMEs), autonomous complexity mapping Week 2, deliverable is a data-driven engagement proposal with real acceleration numbers. Andrew had already begun describing this approach himself before Colin pitched it, which signals alignment.

05

New Requirements Surfaced

Seven new capabilities were requested during the Q&A that were not part of the original Track B demo scope. Five are blocked by the same thing: no access to Sephora's production systems.

Needs Access
Data Reconciliation (EDW vs Databricks)
Raised by: Vishal Sharma, clarified by Grishi
Validate that the output on Databricks matches the legacy EDW source after migration. Colin confirmed this is already built, waiting to be wired in once access is available.
Needs Access
Migration Detection (what already exists in Databricks?)
Raised by: Maher Burhan, confirmed by Andrew
Can the agent scan Databricks and determine which tables have already been migrated? Colin described the knowledge graph exploration approach. This is essentially the exploration SOW itself.
Needs Access
Schema Mismatch Handling
Raised by: Grishi
Target Databricks schema will not match legacy EDW schema 1:1. Colin explained two paths: auto-resolve if non-breaking, flag for human if breaking. Can expand the pipeline with new agents for data migration.
Needs Access
Live Data Sampling for Column Mapping
Raised by: Malika Seth
Would the LLM read actual data (e.g., 10 records) to validate column mapping? Colin recommended two paths: pre-cache during exploration (recommended), or give LLM read-only tools for live queries as fallback.
Needs Access
Cognos MCP Connectivity
Ongoing from prior meetings
Live integration with Cognos for automated report extraction. Java SDK exists. Requires Cognos environment access. Natural first deliverable of the production engagement.
Can Address Now
Source-to-Target Mapping (Manual Feed)
Raised by: Maher Burhan
Can Sephora provide the source-to-target mapping if they already have it? Colin confirmed yes. This is a configuration option, not a new capability.
Can Address Now
ADF as Alternative Orchestration
Raised by: Vishal Sharma
Can Azure Data Factory replace FastAPI for orchestration? Colin confirmed it is adaptable. This is a platform preference, not a new capability.
