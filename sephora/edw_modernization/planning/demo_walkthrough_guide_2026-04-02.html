01

Starting the Application

Get the demo running on your machine before the call starts. Allow 2-3 minutes for everything to come up.

# Make sure you have the latest code
cd ~/programming/sephora_edw-1
git pull

# Start Postgres (for checkpointing + run history)
docker compose up -d

# Install dependencies (if anything changed)
poetry install

# Verify .env has Azure AI + LangSmith keys
cat .env

# Start the application server
PYTHONPATH=src uvicorn etl_agent.api_server:app --host 0.0.0.0 --port 8100

# Open in browser
http://localhost:8100
Before the call
Do a full test run to warm the pipeline and verify everything works. Check that the run history page shows previous runs (Saurav added sample entries). Have LangSmith open in a separate browser tab.

3
Pipeline Execution
This is the main event. The pipeline runs for approximately 9-10 minutes.

What is on screen: A DAG visualization with nodes that light up as each stage completes. Below it, an execution log with live spinners showing what each agent is doing in real time (e.g., "Parsing XML structure... 4s", "Analyzing stored procedure logic... 12s"). A progress bar tracks overall completion.

What the application is doing: The FastAPI server runs the LangGraph pipeline asynchronously (POST /run). A LiveActivityHandler callback tracks every tool call and LLM invocation. The frontend polls GET /status and GET /live to update the DAG, execution log, and progress bar in near-real-time.

The pipeline stages in order:

  • Orchestrator (Sonnet) — loads and classifies all 9 files, identifies the inventory_periodic_daily pipeline from the target Scala app
  • Gate 1 — verifies all artifacts loaded, DataStage XML present, targets found, run ID generated
  • Three parallel parsers (2 at a time due to concurrency limit):
    • DataStage Parser (Opus) — extracts job stages, embedded SQL, column definitions from XML via Python/sqlglot, then one Opus synthesis call
    • SQL Interpreter (Sonnet) — extracts tables, joins, CTEs, temp tables from stored procedures via sqlglot, then parallel Sonnet batch for business rules
    • DDL Parser (Sonnet) — extracts table schemas with exact decimal precision via sqlglot, then parallel Sonnet batch for semantic analysis
  • Gate 2 — verifies parser outputs, cross-checks deterministic vs LLM table extraction, grounds column names in source files
  • Pattern Mapper (Opus) — single call, creates the SQL-to-Spark translation dictionary from all parser outputs plus target examples
  • Gate 3 — verifies mapping references actual tables and includes source refs, type mappings, column mappings
  • Spark SQL Generator (Opus + Sonnet) — plans all stages with dependency graph, generates independent stages in parallel (Sonnet), then dependent stages (Opus)
  • Gate 4 — sqlglot syntax check, DDL table and column coverage, source table grounding, truncation detection, then LLM business logic check at 75% threshold
  • Config Generator (3x Sonnet) — three parallel calls producing pipeline YAML, deployment YAML, and create.HQL by copying target templates exactly
  • Gate 5 — YAML validity, HQL has CREATE TABLE, columns grounded in target, sourceFiles grounded in source, then LLM cross-reference check
  • Schema Validator — deterministic cross-checks, assembles all gate LLM findings, computes overall score
What to say (as parsers start)
"Three agents are now working simultaneously. The DataStage parser is analyzing the XML structure to extract job stages and embedded SQL. The SQL interpreter is analyzing your stored procedures for business logic, flag calculations, and aggregation patterns. And the DDL parser is mapping every column with exact precision from your table definitions. Each one uses sqlglot for deterministic extraction first, then the LLM fills in semantic understanding."
What to say (as gates pass)
"Each of these gates is a deterministic checkpoint. Zero LLM cost. It verifies that the tables the agents found actually exist in your source files, not hallucinated. If anything fails, the gate routes the agent back to retry with the specific error, so it knows exactly what to fix."
What to say (during pattern mapper, the slowest step)
"This is the pattern mapper. It is reading everything the three parsers found, plus your target Scala application and YAML templates, and building a complete translation dictionary. Surrogate keys to business keys, SQL Server types to Databricks types, temp table patterns to intermediate views. This is a single Opus call because it needs the full context to get the mappings right."
If throttling makes it slow
"We submitted a quota increase request on Azure AI Foundry earlier today. In production, this runs faster with higher concurrency. The architecture supports as many parallel agents as you want. Right now we are running at conservative concurrency to stay within our demo quota."

Gate Results
Expandable dropdowns showing pass/fail for each gate

What is on screen: Collapsible sections for each gate. If the score is 100%, the section is closed. If there are issues, it stays open and shows the specific checks that passed or failed, with details.

What the application is doing: The Schema Validator node assembles results from all five gates into a structured validation_result with four sections (Spark SQL, HQL DDL, Pipeline YAML, Deployment Config). Each section has named checks with pass/fail and detail strings. The frontend renders these as expandable cards.

What to say
"Every check here is something you can trace. Column mapping shows how many columns from your DDL were found in the generated SQL. Source table grounding confirms every FROM and JOIN reference in the SQL actually exists in your original files. If something was hallucinated, it would be flagged right here before it ever gets to a human."
Do
Open one or two gate sections to show the detail. Point out specific checks like "DDL table coverage: 8/8 referenced" or "sqlglot syntax: 22/24 statements parsed." These are concrete numbers Grishi will appreciate.
5
Review Step
The human-in-the-loop moment. This is where the demo lands hardest.

What is on screen: Overall validation score (e.g., 97.3%), a list of flagged issues categorized as warnings and infos (never errors), preview of generated SQL with syntax highlighting (highlight.js), and buttons for AI Review, Auto-Fix, Approve, and Reject.

What the application is doing: The pipeline has reached the human_approval node, which uses LangGraph's interrupt() to pause execution and wait for a human decision. The frontend displays the validation_result and generated outputs. The pipeline is paused, not stopped. Approving resumes it to completion.

The Confidence Score

This is the most important talking point of the entire demo.

What to say
"97% confidence does not mean 97% accuracy. It means 97% of the conversion was handled automatically with full validation. The remaining 3% are items flagged for a quick human review. They are not errors. The orchestrator retries until actual errors resolve. What you see here are things like implicit decimal type casts and performance considerations. An engineer would look at these and say, 'yes, I see this all the time.' That is exactly where the human-in-the-loop value is."

The Issues List

Issues are carefully categorized as warnings and infos, never errors. Saurav designed this deliberately. If an actual error existed, the orchestrator's retry loop would have caught it and sent the agent back to fix it before reaching this screen.

What to say about issues
"Notice these are all warnings and infos, not errors. An implicit decimal cast is worth flagging so an engineer can confirm the precision is correct. A performance advisory about worker count is worth knowing about for production tuning. These are insights, not failures. And this is a team that does reporting. The moment you show them insights, you have their attention."

AI Review Button

Do
Click "AI Review." This sends all validation details, the generated SQL, the target Scala application, and the gate LLM findings to the review agent for an audit. The agent has tools to read the SQL, read the Scala, read the validation results, and it produces a detailed analysis. The output is descriptive enough that a data engineer can act on it in one read.
What to say while AI Review runs
"This is an independent audit. The review agent gets the full context: the generated SQL, the original Scala application, the validation results, and the gate findings. It checks for false positives in the validation and looks for anything the deterministic checks might have missed. Think of it as a second set of eyes before an engineer even looks at it."
6
Auto-Fix
Like Claude Code for your pipeline

What is on screen: After the AI Review completes, the Auto-Fix button becomes available. Clicking it triggers the review agent to make surgical edits to the generated SQL based on the review findings.

What the application is doing: The review agent (review_agent.py) has tools: read_sql, read_scala, read_validation, patch_view (replace a specific view's SQL), and run_validation (re-run checks). It maintains a single conversation thread, reads the SQL as a file, identifies what needs to change, and makes targeted edits. It does not regenerate everything. It patches specific views.

What to say
"This is like Claude Code for your pipeline. The agent treats the generated SQL as files. It has tools to read them, edit specific views, and re-validate after changes. It does not regenerate everything from scratch. It makes surgical corrections based on the review findings. After the fix, it runs validation again to confirm the issues are resolved."
Do
Click Auto-Fix. Wait for it to complete. The validation will re-run and the score should improve.
Do not
Do NOT click "Reject and Retry." This works on the backend (routes back to Spark SQL Generator for full regeneration through all downstream nodes), but the DAG animation breaks on retries because the frontend is vanilla HTML/JS without a framework. The execution log will show it running correctly, but the DAG will not light up properly. Use Auto-Fix instead.

The payoff. Show the output files.

What the application is doing: Clicking Approve sends POST /approve with {"approved": true}. The LangGraph pipeline resumes from the human_approval interrupt, the approval count increments, and the pipeline reaches END. The run is saved to PostgreSQL. Generated files are available via GET /output/{key}.

Click Approve. Then download each file individually and show them:
  • Spark SQL — syntax highlighted, all CREATE TEMP VIEW statements with the transformation logic
  • Pipeline YAML — matches the AggregationApplication framework format exactly

  • Create.HQL — the Databricks table definition

"These are the three config files Sergey described. Pipeline YAML, deployment YAML, and HQL. They plug directly into your existing AggregationApplication framework. No new code, no notebooks, no new framework for your support team to learn. The Spark SQL contains the transformation logic. Everything here is production-ready structure. Your team would review it, approve it, and commit it to the repo."

LangSmith (If Asked)
The technical deep-dive tool. Open only when someone wants to go deeper.

What is on screen: LangSmith's waterfall view showing every API call, tool invocation, and node transition for the pipeline run. Blue nodes are LangGraph stage level. Orange nodes are individual LLM calls. You can click any node to see its input and output.

What the application is doing: LangSmith receives trace data from the LiveActivityHandler callback via the LangSmith SDK. It is not real-time (1-2 minute delay), but once the run completes, the full trace is available. You can switch between "trace" and "thread" views, and toggle waterfall mode.

What to say
"This is the full execution trace. Every API call, every tool invocation, every node transition. You can see the parallel parsers running simultaneously. You can click any stage to see exactly what went in and what came out. No question you have about how this works cannot be answered from this view."
When to open LangSmith

Do not open LangSmith proactively. Wait for someone to ask a question that the dashboard cannot answer. If Maher asks about the agent architecture internals, parallel execution, or how a specific stage processed the data, that is when you switch to LangSmith. It is the safety net for any technical question.

03

Known Issues and Workarounds

Reject and Retry button
Backend works fine (routes back to SQL Generator for full regeneration). But the DAG animation breaks on the retry path because the frontend is vanilla HTML/JS. The execution log shows everything running correctly. Use Auto-Fix instead.
ZIP download
Not working. Individual file downloads work fine. Download one at a time.
Duration timer
Includes human wait time. If you spend 15 minutes talking during the review step, the total duration will show an inflated number. If asked, explain: "That includes the time we spent reviewing together. The actual pipeline processing time is visible in the LangSmith waterfall."
API throttling
At concurrency 4, Azure AI Foundry may return 529 (Overloaded) errors. The pipeline handles this with exponential backoff (up to 5 retries, starting at 2 seconds). It will not crash, just run slower. A 4-5X quota increase was requested.
Configuration dropdowns
Target platform, model selection, retry count are visual placeholders. They look good but do not do anything. Do not interact with them during the demo.
Advisory text rendering
The auto-fix advisory section has a badge/label and "Click to view" but the text body may not render fully. The data is there. If it looks incomplete, pivot to the AI Review output which has the same findings in full detail.

Common Questions and Answers

Anticipated questions from Andrew, Grishi, and Maher based on their concerns from prior meetings and Maher's technical background.

"You showed one pipeline. How does this work across hundreds of different types?"
The orchestrator identifies pipeline types from target examples. The knowledge base accumulates approved patterns from each run. Once the first few pipelines of a type are approved, similar ones move through with minimal human intervention. The exploration phase we are proposing is specifically designed to map the full complexity distribution so we can quantify that acceleration.
Likely from: Andrew (scalability) or Maher (generalization)
"What happens when the agent encounters something it cannot translate?"
The deterministic gates catch failures before they propagate. If a gate fails, the producer retries with the specific error context injected into state. After max retries (configurable, default 2), the pipeline proceeds and flags the issue for human review. Nothing gets silently dropped. Every decision is auditable in LangSmith.
Likely from: Maher (error handling)
"Can you trace every transformation back to its source?"
Yes. Source grounding is built into every gate. The validators check that generated table names, column names, and source file references actually exist in the original artifacts. LangSmith provides the full execution trace. Nothing is a black box.
Likely from: Maher (auditability, lineage)
"How does the Cognos MCP work technically?"
Cognos has full Java and .NET SDKs going back to version 10, post-2003. The MCP server wraps the SDK to give agents programmatic access to report metadata, embedded SQL, and RBAC information. For this demo, we used exported files because live MCP requires access to your Cognos environment. The exploration phase would include validating the connector against your specific version.
Likely from: Maher (deep Cognos expertise, 8 years at IBM)
"Does the output actually fit our existing framework?"
The config generator copies the target templates Sergey's team provided and only changes values where the pipeline data requires it. The gate validators check that YAML token names, HQL column types, and deployment config patterns match the target examples. The output is pipeline YAML, deployment YAML, and HQL. No notebooks, no new framework.
Likely from: Grishi or Maher (framework fit)
"What model are you using and how do you handle token costs at scale?"
Claude Opus for heavy reasoning (pattern mapping, SQL generation planning, dependent stages). Claude Sonnet for lighter tasks (orchestration, independent stages, config generation). Both through Azure AI Foundry, which is significantly cheaper than the commercial API. Deterministic gates are zero LLM cost. The architecture minimizes token usage by doing as much as possible with Python and sqlglot before involving the LLM.
Likely from: Maher (production viability)
"How long does this take per pipeline?"
This run takes approximately 9-10 minutes end to end on our current demo quota. In production with higher concurrency and optimized quotas, that comes down. But the real acceleration is not per-pipeline speed. It is that the same pipeline handles the next hundred with the same effort because the knowledge base accumulates patterns. The work to do 100 is about the same as 1,000 once patterns are established.
Likely from: Andrew (timeline compression)
"Does it beat what we are doing with Claude today?"
Your team uses Claude for one step in the workflow: SQL conversion, getting about 30% efficiency. This pipeline automates the entire workflow: parsing, interpretation, mapping, generation, validation, and review. The efficiency gain compounds across all steps, and the deterministic gates ensure consistency that manual Claude interactions cannot provide.
Likely from: Grishi (30% baseline)
"What about table data migration from SQL Server to Databricks?"
Table data migration is absolutely within scope. It is the other half of what we showed today. This demo covers ETL pipeline conversion. The data migration layer uses the same agent architecture but requires access to the source SQL Server and target Databricks to build that pipeline. That is part of the exploration phase we are proposing.
Likely from: Andrew (scope expansion from Set 04)
"What is the next step?"
Two options. One, we deploy this on Azure with secure credentials so your team can try the workflow directly this week. Two, and this is what we recommend, a focused two-week exploration. Week one, your technical team walks us through the full landscape. Week two, we build the complexity map autonomously and give you a data-driven engagement proposal with real acceleration numbers.

If Something Goes Wrong

The safety net is LangSmith

LangSmith has the complete pipeline data for every run. If the dashboard has a rendering issue, if a section does not load, or if something looks wrong on the frontend, pivot to LangSmith. It can answer any technical question and show any pipeline data that the frontend cannot display.

If the pipeline crashes entirely (unlikely but possible with throttling), you can show the most recent completed run in LangSmith and walk through the waterfall view. The data is there even if the frontend is not cooperating.

If something visually glitches
"Let me show you this in more detail." Switch to LangSmith. Do not apologize for the UI. Reframe it as going deeper.
If throttling makes it very slow
"We are running on demo-tier quotas right now. A quota increase was submitted earlier today. In production this runs at full concurrency. Let me show you a completed run while this one finishes." Switch to a previous run in the run history, or show the LangSmith waterfall of a prior run.
