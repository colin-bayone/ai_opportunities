What They Want to See

From Malika's scope email (written on behalf of the technical team) and direct statements from Andrew and Grishi across three meetings.

  • Agent-driven orchestration across multiple tools (not one agent doing one thing)
  • End-to-end workflow automation (extraction, interpretation, and downstream execution coordinated by agents)
  • Task sequencing and dependency handling (how the pipeline manages what runs when)
  • Parsing of their actual DataStage XML, stored procedures, and views
  • Output that fits their existing framework (Spark SQL + YAML configs for AggregationApplication, not notebooks)

04

Demo Flow

Seven phases. Each maps to something Andrew or Grishi specifically asked for. Open with the dashboard, not slides.

Opening
2 min
Maps to: Credibility, "built for you"

Do NOT open with slides. Open with the dashboard. One sentence: "Everything you see was built specifically around your DataStage jobs, stored procedures, and source schemas over the last two weeks." Then start the pipeline.

Phase 1: Load Their Actual Files
2 min
Maps to: "Can it handle our specific data?"

Click "Load Demo Files" (their 9 actual files preloaded). Point out: these are Sephora's files. DataStage XML, stored procedures, DDLs, views, Databricks schemas, target framework examples. Click "Start Pipeline."

Phase 2: Watch the Execution Log
3-5 min
Maps to: "End-to-end workflow automation" and "task sequencing"

Live spinners show each agent step. Call out parallel parsers: "Three agents working simultaneously. DataStage parser analyzing XML, SQL interpreter extracting business rules, DDL parser mapping column types." When gates pass: "Deterministic checks, zero LLM cost, verifying tables exist in your source files."

Phase 3: Walk Through Gate Results
2-3 min
Maps to: "How does it handle quality?"

Open gate dropdowns. Show pass/fail checks: column mapping, source table grounding, SQL syntax. "LLM checks only run after deterministic checks pass, which saves tokens." This is for Grishi. She will want specifics.

Phase 4: Review Step
3-5 min
Maps to: "Human-in-the-loop" and "confidence score"

"97% confidence means the remaining items are not errors. They are small conversion items flagged for human review." Show the issues: warnings and infos, never errors. Click "AI Review" to show the audit. Explain: "An engineer would look at these and say 'I see this all the time.' That is the human-in-the-loop value."

Phase 5: Auto-Fix
2-3 min
Maps to: "Can the system self-correct?"

Click Auto-Fix (NOT reject-and-retry). "This is like Claude Code for your pipeline. It treats generated SQL as files and makes surgical edits based on review findings." This eliminates the manual iteration Grishi described.

Phase 6: Approve and Download
2 min
Maps to: "Does the output fit our framework?"

Click Approve. Download individual files. Show: Spark SQL (syntax highlighted), Pipeline YAML, Deployment YAML, Create.HQL. "These are the three config files Sergey described. They plug directly into your existing framework."

Phase 7: LangSmith (if asked)
Optional
Maps to: Technical deep-dive for Andrew/Grishi

If anyone asks "how does it work under the hood?" open LangSmith. Waterfall view shows every API call, every node, parallel execution visually. "No question you have cannot be answered from this."

Talking Points by Concern

Their Concern What to Say
"Is this real?" (Andrew) The pipeline just processed your actual DataStage XML, stored procedures, and schemas. The output plugs into your existing framework.
"Does it beat Claude alone?" (Grishi, 30% baseline) Your team uses Claude for one step: SQL conversion. This automates the entire workflow. The efficiency gain compounds across all steps, and the deterministic gates ensure consistency that manual Claude interactions cannot.
"What about old software versions?" Cognos and DataStage both have Java/.NET SDKs going back to 2003. For this demo we used exported files. In production, MCP connectivity replaces the manual export.
"Does it fit our framework?" (Sergey) Pipeline YAML, deployment YAML, and HQL. No Scala code, no notebooks. These work with your existing AggregationApplication framework.
"Can it scale?" (Andrew) Once agents and patterns are in place, 100 reports is the same work as 1,000. The knowledge base learns from each approved run.
"What about Cognos MCP?" The MCP connector is the natural next step. A Java hook exists. Once we have access to your environment, the pipeline runs fully end-to-end.
"What is the next step?" We can deploy on Azure with secure credentials so your team can try the workflow. Time-bound access, your data already loaded. Or we schedule a deeper session.
"How does the agent architecture work technically?" (Maher) LangGraph StateGraph with 15 nodes and 5 deterministic gates. Parsers use sqlglot for AST-based extraction before involving the LLM. Source grounding checks verify every generated name exists in the original artifacts. Retry loops pass specific error context back to the producing agent. LangSmith has the full execution trace.

Things to Avoid

Do NOT click "Reject and Retry" (DAG animation breaks, use Auto-Fix instead)

Do NOT mention cost per run or token counts (they will multiply by 6,000 reports)

Do NOT oversell MCP (requires environment access, which means a contract)
Do NOT dwell on the UI if they want technical depth (pivot to LangSmith)
Do NOT use the word "product" (this is a custom solution built for them, not an off-the-shelf tool)
Do NOT oversimplify the architecture (Maher has deep technical expertise and will notice handwaving)
