What Was Demonstrated

A multi-agent pipeline built specifically around Sephora's DataStage jobs, stored procedures, source schemas, and target framework over the past two weeks.

Agent Orchestration
Multiple specialized agents coordinating across parsing, interpretation, mapping, generation, and validation stages with deterministic quality gates between each step.
DataStage ETL Conversion
DataStage XML, SQL Server stored procedures, DDLs, and views parsed and converted to Databricks-compatible Spark SQL and YAML configuration files.
Deterministic Quality Gates
Every agent output is validated by deterministic checks (zero LLM cost) before proceeding. Source table grounding, syntax validation, column mapping, and business logic verification.
Framework-Compatible Output
Generated output follows the existing AggregationApplication framework format: pipeline YAML, deployment YAML, and HQL table definitions. No new frameworks introduced.
Human-in-the-Loop Review
AI-powered review identifies conversion items for human validation. Auto-fix applies surgical corrections. The system learns from approved patterns for future runs.
Parallel Execution
Independent pipeline stages run concurrently. Three parsers execute in parallel. SQL generation plans a dependency graph and generates independent stages simultaneously.
Key Result

The pipeline processed Sephora's actual source materials end to end and produced output that conforms to the existing production framework. The approach is proven. The remaining question is how to apply it across the full environment.

Quality Gates

GateNameWhat It Checks
G1 Orchestrator Gate All source artifacts loaded, DataStage XML present, target examples loaded, all artifacts have content, run ID generated
G2 Parser Gate DataStage parser returned stages and embedded SQL, SQL interpreter produced results, DDL schemas parsed, column names grounded in source files, cross-check between deterministic and LLM table extraction
G3 Mapper Gate Pattern mapping is non-empty, mapping references actual source table names from DDL schemas, mentions source references, type mappings, and column mappings
G4 SQL Gate SQL is substantial (5,000+ chars), sqlglot syntax check (50%+ statements parse), DDL table coverage (30%+), column coverage (20%+), source table grounding (FROM/JOIN refs exist in source artifacts), truncation detection (end-of-file and per-view), HQL column match. Then LLM check for business logic preservation and column existence against the Scala source
G5 Config Gate Pipeline YAML valid and substantial (200+ chars), deployment YAML valid (100+ chars), HQL has CREATE TABLE, HQL columns grounded in target, YAML sourceFile names grounded in source. Then LLM check for token name matching, column type consistency against SQL CASTs, and deployment config structure

Agent Roles

AgentModelWhat It Does
Orchestrator Sonnet Loads all source files, classifies each by type (DataStage XML, stored procedure, DDL, view, schema, Scala app, YAML, HQL), identifies which pipeline to build from target examples, generates run ID
DataStage Parser Opus Parses DataStage XML: extracts job stages, transformations, embedded SQL, and column definitions using Python and sqlglot deterministically, then one Opus call to synthesize findings into structured analysis
SQL Interpreter Sonnet Analyzes stored procedures and views: extracts tables, joins, CTEs, and temp tables via sqlglot, then parallel Sonnet batch calls for business rule extraction (flag calculations, aggregation patterns, temporal logic)
DDL Parser Sonnet Parses DDL files and views: extracts table names, column names with exact decimal precision, nullable constraints, and primary keys via sqlglot, then parallel Sonnet batch calls for semantic analysis
Pattern Mapper Opus Single Opus call. Reads all parser outputs plus target Scala and YAML examples. Produces translation dictionary: surrogate keys to business keys, SQL Server types to Databricks types, temp tables to intermediate views, CASE WHEN patterns, GETDATE to current_date
Spark SQL Generator Opus + Sonnet Three-phase generation: (1) Opus plans all stages and builds a dependency graph of independent vs dependent stages, (2) Sonnet generates independent stages in parallel, (3) Opus generates dependent stages with Phase 1 output as context. On retry, a single Opus call with previous SQL plus errors for targeted fix
Config Generator 3x Sonnet Three parallel Sonnet calls, one per file: pipeline YAML, deployment YAML, and create.HQL. Each copies the target template structure exactly, only changing values where the generated Spark SQL requires it
Schema Validator None Zero LLM calls. Cross-references all generated outputs against each other and against source artifacts. Assembles gate LLM findings from upstream. Computes overall validation score. Produces the result for the review screen
Human Approval None Presents validation score and flagged issues for human review. Approve proceeds to completion. Reject routes back to the Spark SQL Generator with error context for targeted correction
Design Principle

Deterministic first, LLM second. Every gate runs Python and sqlglot checks at zero LLM cost before proceeding. LLM-based quality checks only run after deterministic checks pass. This minimizes token usage while maintaining high validation standards.

Source grounding at every gate. Generated table names, column names, and source file references are checked against the original artifacts. If a name does not exist in the source data, the gate flags it as potentially hallucinated before it propagates downstream.

Why This Approach

The most responsible path to an accurate engagement plan.

Without the exploration, any engagement estimate is a flat calculation: the total number of reports and data flows multiplied by the time required for one conversion. That approach does not account for the acceleration that agents provide through pattern reuse, knowledge base learning, and parallel execution.

The agent-driven flywheel only takes effect when there is a clear picture of the overall complexity distribution, both in terms of pipeline types and their relative complexity. Some pipelines will closely resemble the one demonstrated today and can move through the system with minimal human intervention. Others will present unique patterns that require additional configuration.

This exploration produces the data needed to distinguish between those categories and provide acceleration estimates grounded in the actual environment rather than assumptions. It also seeds the knowledge base with the real complexity distribution, so the agent flywheel begins accelerating from the first day of the full engagement rather than learning from scratch.

Additional Benefit

The exploration also validates the Cognos MCP integration, which was identified as a priority during the technical deep dive but could not be demonstrated without access to the Cognos environment. With access, BayOne can confirm the connector works and scope the full end-to-end automation that includes automated report extraction, conversion, and deployment.

Security and Data Handling

Key Security Properties

All model inference runs through Azure AI Foundry. No data is sent to commercial API endpoints. Azure AI Foundry provides the same enterprise security guarantees as Azure Active Directory, Outlook, and Teams.

Source data stays in Azure Blob Storage. Sephora's DataStage XML, stored procedures, and schema files are uploaded to a private storage account. Generated outputs are written back to the same storage account.

Time-bound access. Demo environments are provisioned with credentials that expire automatically. The server shuts down after a defined period. Access can be restricted to specific individuals.

Read-only exploration. The proposed exploration phase requires only read access to Sephora's systems. No writes, no modifications, no data extraction beyond what is necessary for complexity mapping.
