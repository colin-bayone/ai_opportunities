## What We Got Wrong (or Didn't Anticipate)

### 1. The entire problem definition changed
This is the single biggest change across all sets. Sets 01 and 02 framed the work as: "EPNM has functionality that EMS does not. Convert it." Set 03 reframed it as: "EMS has the functionality but with a different UX. Give customers the option to use the classic EPNM experience." These are fundamentally different engineering challenges. The first is a build problem. The second is a presentation problem.

### 2. "Missing reports" is no longer the starting point
Set 02: Selva specifically identified missing reports as a concrete POC target. Set 03: The POC targets are faults and inventory, screens that already exist in EMS. Missing reports were not mentioned.

### 3. The toggle UX concept was never discussed before
Sets 01 and 02 had no mention of a toggle, classic view, or dual-experience approach. This concept was introduced entirely in Set 03 by Selva, using the Outlook and banking app analogies.

### 4. Backend is now explicitly off-limits
Sets 01 and 02 assumed significant backend work. Set 03 established a hard boundary: the EMS backend stays. Only minor API-level adjustments for filtering differences are acceptable. Selva warned that backend changes could destabilize the critical July release.

### 5. The business driver shifted from "fill gaps" to "ease adoption"
Sets 01 and 02: customers need functionality that is missing. Set 03: customers have the functionality but resist the new UX because operators are trained on the old interface for 10+ years. The goal is now about reducing migration friction, not building missing features.

## What We Got Right

### 1. AI acceleration is the right approach
The methodology Colin presented in Sets 01 and 02 (Claude Code, LangGraph, Playwright) remains relevant. The scope change actually makes it MORE suitable: a UX overlay is more amenable to pattern-based generation than full-stack vertical work.

### 2. The exponential decay / flywheel concept applies
Colin's explanation of first-screen front-loading in Set 03 reinforced the acceleration mechanism from the POC proposal. The scope change makes this even more powerful because UX toggle screens are more similar to each other than full-stack vertical conversions.

### 3. Colin running it solo for POC is the right model
Confirmed again in Set 03. Selva's team is in India and on critical platform work. The independent-work-with-checkpoints model stands.

### 4. Selva is the operational counterpart
Set 02 showed Selva emerging as more active. Set 03 confirmed it: Guhan departed early and Selva took full operational ownership, scheduling team sessions, creating WebEx space, and identifying local resources.
