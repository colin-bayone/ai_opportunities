Recruiter Guide

Automation Engineer

On-site San Jose, CA | Client-facing | CI/CD + Integration Focus
Helpful for pre-screening: If your screening call is on Teams, please enable transcription and share the transcript afterward. This helps our technical team review candidates more effectively. Not required, but appreciated!
Must-Have Skills
  • 4+ years software engineering, 3+ years CI/CD
    Has built pipelines and automation, not just used them as a developer
  • Apache Airflow experience
    Airflow schedules and runs tasks in sequence. Should have built workflows in it, not just used dashboards
  • Jenkins experience
    Should have configured pipelines or built integrations, not just clicked "run"
  • Python backend development
    Comfortable building APIs, data pipelines, and backend services
  • SQL and database skills
    Can design tables, write complex queries, work with PostgreSQL
  • Integration experience
    Has connected multiple systems - pulling data from one, transforming, pushing to another
  • Strong communicator
    Clear writing, comfortable explaining technical systems to others
  • AI pair programming proficiency
    Actively uses Claude Code, Cursor, or similar agentic AI tools daily. Not just autocomplete (Copilot) - should be comfortable with AI that writes and edits code autonomously
  • Experience working with AI systems
    Has built integrations with AI features, or worked alongside AI engineers. Understands how AI components fit into larger systems
Nice-to-Have Background
  • Bazel experience
    Google's build system, used for large codebases. Less common but valuable
  • Grafana dashboards
    Built monitoring dashboards to visualize system health
  • GitHub APIs / webhooks
    Automated things based on GitHub events (PRs, commits, etc.)
  • On-prem deployment
    Deployed systems in corporate data centers, not just cloud
  • Platform engineering background
    Worked on internal tools that help other developers be productive
  • Working with legacy systems
    Comfortable integrating with old, poorly-documented systems
Strong Signals
  • Built integrations between multiple CI/CD systems (not just used one tool)
  • Has worked on "developer productivity" or "platform engineering" teams
  • Experience at companies with 100+ engineers (enough CI/CD complexity)
  • Comfortable with incomplete documentation - can figure things out
  • Enthusiastic about AI tools - can describe specific ways Claude Code or Cursor changed their workflow
  • Has debugged issues that span multiple systems
  • Has worked on projects that involved AI components or alongside AI engineers
Warning Signs
  • Only used CI/CD as a developer - never built or configured pipelines
  • Can't explain what Airflow does or when you'd use it vs other tools
  • Only worked in brand-new environments - no legacy integration experience
  • Uncomfortable when things aren't well documented
  • Doesn't use AI coding tools, or only uses basic autocomplete (Copilot alone)
  • Vague about what they personally built vs what the team built
  • No experience working with people outside their immediate team
Screening Questions

Record all screening calls. These questions reveal depth without requiring technical evaluation. Note answers verbatim for technical review.

"Describe an Airflow DAG you built. What did it do and why did you choose Airflow?"
Weak Answer
Can't describe a specific DAG. Gives vague answers like "it ran some jobs." Doesn't know why Airflow was chosen vs other options.
Good Answer
Describes a specific workflow with details - what tasks ran, in what order, what happened if something failed. Can explain why Airflow made sense for that use case.
"Tell me about a time you had to integrate with a system that had poor documentation. How did you figure it out?"
Weak Answer
Can't recall a specific situation. Or says they just asked someone else. Or got frustrated and gave up.
Good Answer
Tells a specific story - how they explored the system (reading code, testing APIs, tracing logs), what didn't work, how they eventually figured it out.
"What's the difference between Airflow and Jenkins? When would you use each?"
Weak Answer
Can't explain the difference, or thinks they're the same. Gives a textbook definition without real understanding.
Good Answer
Explains Jenkins is for CI/CD (triggered by code changes) while Airflow is for scheduled workflows (time-based or dependency-based). Gives examples of when each makes sense.
"Tell me about a time you had to debug an issue that involved multiple systems. How did you track down the problem?"
Weak Answer
Can't recall a specific situation. Or just says "I looked at the logs" with no detail.
Good Answer
Walks through a specific debugging journey - which systems, how they narrowed it down, what tools they used, how they confirmed the fix worked.
"What AI coding tools do you use, and how have they changed the way you work?"
Weak Answer
Doesn't use AI tools, or only mentions basic autocomplete like Copilot. Can't describe specific ways it changed their workflow. Dismissive about AI assistance.
Good Answer
Uses Claude Code, Cursor, or similar agentic tools. Gives specific examples of how it changed their work - faster debugging, writing tests, exploring unfamiliar codebases. Enthusiastic about the productivity gains.
"Can you explain [most complex thing on their resume] to me like I'm not technical?"
Weak Answer
Uses jargon, gets frustrated, or can't simplify it.
Good Answer
Uses plain language and analogies. Checks if you're following. Patient and clear. (You can evaluate this directly.)

What to Look for in Work History
  • Company size 100+ employees
    Smaller companies often don't have complex CI/CD - not enough scale to need it
  • Built automation, not just used it
    Many developers use CI/CD but never configured or built pipelines themselves
  • Integration or platform team experience
    Shows they've connected systems together, not just worked in one silo
  • Experience with messy/legacy systems
    Our client has mature enterprise systems - not everything is new and clean
  • Some client or stakeholder interaction
    This role is 15% client-facing - they need to communicate with external teams
