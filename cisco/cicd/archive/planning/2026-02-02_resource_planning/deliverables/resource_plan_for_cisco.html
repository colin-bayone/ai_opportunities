Executive Summary

BayOne proposes a year-long engagement to improve Cisco's NX-OS CI/CD pipeline through six integrated capability areas. The engagement addresses challenges across the development lifecycle: from lack of visibility into local developer testing, through fragmented cross-pipeline status information, to manual failure triage and limited automation for corrective actions.

The work is phased to build incrementally, with each quarter's deliverables providing the foundation for the next. A team of five begins in Q1, combining on-site and offshore resources. Team composition may adjust over time as we establish working patterns and delivery needs evolve.

Quarterly Roadmap

Q1: Discovery + Developer Box + Branch Health

Discovery Phase (3-4 Weeks)

Before implementation begins, the team will conduct a structured discovery phase requiring approximately 8-12 hours of Cisco team involvement, spread as convenient. The remainder of discovery time is independent research into the codebase and existing systems. Discovery produces a confirmed scope document with refined estimates based on actual system understanding.

A: Developer Box

Telemetry system to capture local test runs with pass/fail, duration, and coverage metrics. Data pipeline to store and surface results. Condition-level coverage tracking.

F: Branch Health

Consolidated dashboard for release leads showing status, failure trends, and blockers. Failure attribution logic. Automated follow-up triggers.

Prerequisites: System access to CAT, DevX, Jenkins, Airflow, Grafana, and relevant codebases. A designated Cisco contact who can unblock administrative hurdles and answer questions.

Q2: Unified Interface / Cross-Pipeline Visibility

C: Unified Interface

Information is currently scattered across multiple systems with no unified view. Engineers submit PRs and have no easy way to monitor progress. This deliverable requires integration with six Cisco systems, each with its own APIs, data schemas, and access patterns:

CAT Gate configurations
DevX Platform PR metadata
Jenkins CI checks
Apache Airflow Long-running jobs
Grafana Analytics
GitHub Source control

API Integration Layer

Connections to all systems with unified authentication and data normalization across different schemas.

Single Pane of Glass

Web interface showing consolidated PR status across all branches with blockers and next actions.

Unified Data Layer

Common data model that abstracts underlying system differences.

Chat Interface

Natural language queries: "Where is my PR?" or "Why did this fail?"

Q3: AI Diagnosis + Coverage Tracking

B: Gate Failures / AI Diagnosis

Log analysis pipeline for Jenkins and Airflow. AI-driven root cause identification. Suggested fixes for human review based on codebase understanding.

D: Coverage Tracking

Condition-level coverage tracking beyond function-level. PR coverage confirmation showing what was tested vs. not tested.

There are 39 gates in the PR validation process. When a gate fails, engineers must manually investigate. AI diagnosis will analyze failure patterns and logs to identify likely causes and suggest fixes for human review. Coverage tracking will confirm that specific code changes in a PR were actually exercised by tests.

Q4: Self-Healing / Automated Corrective Actions

E: Self-Healing

For failures that don't require human judgment, engineers still have to manually intervene. Self-healing means automated retry for transient failures and automated simple corrections for well-defined issues. All actions are taken within a defined governance framework and are logged, auditable, and reversible.

Governance Framework: Before any automated actions can be implemented, a governance framework must be defined collaboratively with Cisco. This includes determining what types of failures can be auto-corrected, what actions the system can take, and how audit and accountability work.

Action Triggers

Integration with Jenkins and Airflow for automated retries and corrections.

Decision Logic

AI-driven determination of which action to take based on failure type and governance rules.

Safety Mechanisms

Limits on automated actions, circuit breakers, and human escalation when confidence is low.

Audit Visibility

All actions logged with full context and surfaced in the unified interface.

Timeline Considerations

E is unique because it requires organizational alignment beyond technical implementation. The governance framework involves policy decisions, stakeholder buy-in, and risk acceptance. We recommend beginning governance conversations in Q2-Q3 in parallel with other work.

Ideally, governance should be established before Q3 to allow sufficient implementation time. Delays in governance decisions beyond the start of Q4 will impact the delivery timeline. This deliverable is a joint responsibility between BayOne and Cisco.
