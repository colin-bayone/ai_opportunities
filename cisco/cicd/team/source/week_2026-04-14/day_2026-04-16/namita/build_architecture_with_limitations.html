// known limitations — click to expand
Subset errors ›
find_earliest_error_log.py uses mtime-based file selection and a backward scan that stops at ARCH lines — it captures only a subset of available error context. bzl_error_report.py similarly relies on log-shape heuristics and regex patterns that may miss unusual compiler output.
No persistence ›
tools/current_errors.txt and .json exist only inside the checked-out WIT workspace during a single run. There is no database storage of analysis results, error history, or fix attempts across runs — nothing is queryable after the session ends.
No traceability ›
The email includes a git diff but there is no record linking a specific patch to the error block that motivated it. Root cause → fix chains are not tracked, making post-hoc analysis impossible.
make retries all ›
automated_workflow.py rebuilds the full original build_targets list on every retry attempt, not just still-failing targets. This wastes time and obscures per-target progress. The Bazel workflow correctly tracks remaining_targets.
Unused param ›
run_codex_analysis() accepts an error_output parameter in both workflow scripts but never uses it — the Codex prompt does not receive this data, reducing the quality of the AI context.
Hard-coded paths ›
BZL_ERROR_REPORT, the WIT binary path, mail script path, and NODEJS_HOME are all hard-coded absolute paths, making the scripts tightly coupled to a specific environment and fragile to infrastructure changes.
find_latest_build_tag ›
workflow_common.py's tag selection sorts API keys ascending and returns the first matching tag — likely returning the oldest available tag rather than the newest. Worth verifying against the actual API key format before relying on this in production.
