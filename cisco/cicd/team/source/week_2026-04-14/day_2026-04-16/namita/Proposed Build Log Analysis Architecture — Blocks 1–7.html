Block-by-Block Notes

Block 1 — NFS Ingestion decision pending

Reads Bazel CI/CD logs from the NFS mount. Two viable modes: (a) scheduled batches every 15–30 min, simpler and predictable; (b) event-driven / realtime using a filesystem watcher or webhook, lower latency but more moving parts. Proposal: start with 15–30 min batching to match current build cadence, revisit once queue patterns and SLA targets are confirmed.

Block 2 — Parsing & Chunking

Regex/structured parsing when log formats are uniform; NLP-based chunking where formats vary across job types. NLP summarisation reduces chunk size before any Tier 2/3 call to control token cost. Critical given 300K–500K line files. Final parsing strategy pending log-format verification across CI vs CD and composite vs individual files.

Blocks 3 / 4 / 5 — Tiered Error Evaluation

Tier 1 (B3): deterministic rule-based detection using the Bazel error-code registry and regex over error descriptions. Tier 2 (B4): a smaller specialised ML model classifies errors that Tier 1 cannot match. Tier 3 (B5): LLM performs root-cause investigation and fix suggestion only for novel / compound errors. Escalation is explicit; each tier records its confidence.

Block 5.1 — LLM Code Change & PR

Consumes the root cause, fix suggestion and context from B3/B4/B5. LLM produces a code patch, the system branches and commits via Git, opens a PR with root-cause summary and proposed fix, and triggers build verification. A human review gate on the PR is mandatory; PR status (merged / rejected / build pass) flows back into B6 for traceability.

Block 6 — Structured Storage (Star Schema)

Fact table build_analysis_facts (one row per error per build) linked to dimensions dim_builds, dim_errors, dim_repos, dim_branches, and optionally dim_log_files. Persists summaries, suggested fixes, confidence scores, SHA hashes and PR numbers. Full commit diffs and PR comments are queried live, not persisted.

Block 7 — MCP Tool Interface

Exposes the schema to downstream agents and users via MCP tools: query analysis results, trace an error back to a commit/PR, fetch fix suggestions, pull build-failure summaries, and correlate with live Git/GitHub state. This is the primary consumption layer for agentic AI workflows.
