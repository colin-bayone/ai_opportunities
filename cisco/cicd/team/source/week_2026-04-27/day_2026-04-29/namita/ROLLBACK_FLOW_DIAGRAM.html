Incident Analysis and Rollback

Inputs, Mapping, Failure Detection, Dependency Check, Rollback

This flow shows how to move from PR commit data and build metadata to a safe rollback boundary. The key idea is that the failing commit is only the starting point; the real rollback target may be a connected set of PRs that depend on one another.

1

Read PR commits from `diff`

Gather all reviewed commits for each PR and preserve branch, merge, and parent metadata. This answers: which commits were part of review history?

PR -> commits reviewed history
2

Read build data from `fullbuild`

Capture which commits and PRs actually entered a specific CI build. This gives the build change set, not just the reviewed change set.

build -> commits build -> PRs
3

Normalize into one model

Convert all sources into shared structures such as `commit -> PR`, `PR -> integrated commit`, and `build -> change set`.

common schema traceability
4

Map PRs to integrated commits

Classify commit relationships as source, merge, squash, rebased, or promotion commits. This is the core link between review history and repository history.

merge commit squash commit rebased equivalent
5

Parse failed build logs

Identify the first reliable failure signal: a bad SHA, a broken file, a failing stage, or the first bad build range between last good and first bad builds.

failure signal candidate bad commit
6

Map failing commit back to root PR

Use the normalized mapping to find which PR owns the failing integrated commit. This becomes the root PR for rollback analysis.

failing commit -> PR root PR
7

Build the dependency graph

Inspect later PRs and commits for semantic dependencies such as shared files, changed interfaces, config contracts, promoted changes, or downstream consumers.

hard dependency soft dependency promotion flow
8

Choose rollback boundary

Decide whether to revert only the root PR, a connected PR cluster, or the entire range back to the last known good build if the branch is too coupled or noisy.

single PR PR cluster last good build boundary
9

Create rollback recommendation

Produce a report with failed build, failing commit, root PR, dependent PRs, confidence, recommended revert set, and rationale.

rollback candidate set explainable output
10

Execute rollback and validate

Revert through a new rollback PR, rerun CI, verify the failed stage, and confirm no remaining dependent change is left in a broken state.

revert PR CI validation
Decision Rule

Do not rollback only by timestamp

Time proximity is weak evidence. Prefer semantic dependency: shared code paths, schema changes, pipeline contract changes, feature enablement, or promotion from one branch to another.

Key Output

Rollback Candidate Set

  • failed build id
  • first bad build range
  • root failing commit
  • mapped PR
  • dependent PRs
  • recommended rollback boundary
Outcome

Safe Rollback Goal

The objective is the smallest safe revert set that restores a coherent branch state, not the smallest possible number of commits.

Core idea: PR commits tell you review history, `fullbuild` tells you what entered the failing build, logs tell you what broke, mapping ties the failure to a root PR, and dependency analysis tells you whether you can revert one PR or must roll back a larger connected change set.
