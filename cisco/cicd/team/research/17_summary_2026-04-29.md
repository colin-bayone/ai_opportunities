### 2. Namita's rollback flow walkthrough

Namita presented the 10-step incident-analysis-to-rollback flow against the diagram she committed to `skills/webex`. The flow: read PR commits from `diff`, read build data from `fullbuild`, normalize into one model, map PRs to integrated commits, parse failed build logs to identify the failure signal, map the failing commit back to root PR, build the dependency graph, choose rollback boundary, create rollback recommendation, execute and validate. She also walked through the live sample output (five mapped PRs, one unmapped commit, evidence trails per commit).

Colin's coaching on framing for Srinivas:

- **Steps 1 through 4 are universal across any repo, not Cisco-specific.** Deterministic GitHub commands. This is the reusability story Colin wants in front of Srinivas, because Justin's approach is custom-script-per-case.
- **The contrast with Justin's approach.** Justin gives a pruned error log to an LLM with a 3-try cap and gets a syntax-level suggestion (e.g., "missing semicolon"). Justin's "diff" is not actually a diff (no plus/minus); it is just the pruned error log. Justin's path can resolve static Bazel errors but cannot resolve dynamic code issues because there is no plus-minus history to trace. Namita's path is the correct way and addresses the gap Srinivas's current approach has.
- **The PR-to-PR dependency piece is the only missing element in Namita's diagram.** PRs are chronological, so the dependency is simple: by date of last edit, merge, or push. This is the one straightforward step that completes the recursion (single commit touches one PR which has upstream PRs).
- **Deterministic-as-long-as-possible.** Steps 1 through 6 are deterministic. Step 7 (dependency graph) and step 8 (rollback boundary) are where AI gets inserted because the rules become non-deterministic. Saurav added: visually label deterministic boxes versus AI-driven boxes on the diagram so Srinivas can see the architecture at a glance.
- **Do not pre-decide the form.** Colin's principle: when Srinivas says "build a dependency graph," that is prescribing a form rather than describing an outcome. The right framing is the attribute of the outcome (associate PRs, pull related code, link a build failure to a commit and a PR). If a dependency graph satisfies the attribute, fine; if a script or a language model call satisfies it, also fine.

### 3. Srikar's static-vs-dynamic dashboard

Srikar walked through the dashboard with three-month, six-month, and twelve-month filters. The unanswered set is largely dynamic. The build-image category for example is dominated by dynamic items because there is no clear-cut resolution and people do not answer in the chat thread.

Colin's coaching:

- **Add a chart for the answered set too, not just the unanswered set.** Stacked bar or a second graph. This is the natural counterpart of the unanswered chart Srinivas asked for.
- **Have a standalone markdown document explaining the classification methodology.** Committed to the repo so it can be opened in the browser during the call when Srinivas asks "why is this static and not dynamic?"
- **The static-versus-dynamic definition for the dashboard.** A question is dynamic if (a) there is no clear-cut resolution path that has already been answered, or (b) it is the first time the question is being asked (no historical record).

### 4. The FAQ source-of-truth question (Saurav)

Saurav raised: should the FAQ include the build table queries Justin's MongoDB exposes? And the wiki Cisco gave us, is that part of FAQ scope, or only the build state?

Colin's framing for Srinivas:

- The wiki was shared late and most entries are 3-7 years old. Out of scope for this week. **Going forward, the wiki is a way to expand the static set in the analysis.** Needs a validation pipeline before it becomes live (drift control). Phrase it as upcoming work, not Wednesday-deliverable scope.
- The wiki cannot be the canonical FAQ source without a refresh mechanism. Colin will say this on the call.
- Build-state lookups (Justin's MongoDB) are a separate layer from FAQ scope.
