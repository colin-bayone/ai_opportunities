## The Core Principle: Specify Attributes, Not Outcomes

The first principle Colin teaches is the most important one, and it is the principle that most distinguishes his research methodology from the way Namita has been working. He says, "Here's where things go wrong. You can, You could look at the outcome first, or you could look at the path to get to the outcome. The question is always which one is more valuable." He then states the rule directly. "The trick is that unless you know exactly what your outcome is going to look like, you should never specify the outcome. You should specify the attributes of the outcome."

This is a counterintuitive principle for most researchers, who are trained to formulate a hypothesis and then test it. Colin is teaching the opposite. He explains, "If I go here, even if I, you know, wrote some more sentences to be more specific about what that dependency map took shape, I don't actually know what I want yet. I know the general goal that I have, but I also don't quite know how I want to get there yet." The honesty of this admission matters. Colin is the senior technical leader on the engagement, and he is openly acknowledging that for novel research questions he does not yet know what the right answer looks like. The methodology is designed to operate in exactly that condition.

To make the principle vivid, Colin uses a memorable analogy. He says, "If you go online and you try to prove yourself right, you will prove yourself right. If you want to find an article that says chocolate causes cancer, you're going to find it. And the same thing goes with research." He continues, "If you go in and you prescribe a solution before you know the path to that solution, you will find what you're looking for. But will it be correct? That's a different story." The chocolate-causes-cancer analogy is doing important pedagogical work. It captures the failure mode of confirmation bias in a way that is immediately relatable, and it positions over-specified research prompts as a mechanism for producing exactly that failure mode at scale. When a researcher prescribes the solution in the prompt, the language model dutifully finds support for that solution. The output looks authoritative but it is essentially a curated confirmation of the researcher's prior assumptions.

This principle has a direct connection to the GitHub dependency graph problem. Namita and Anupma had been working on the assumption that the dependency graph required custom commit-attribution logic. Colin's view, which he validates through the research methodology he is about to demonstrate, is that built-in tooling and standard git command-line capabilities can accomplish most of what they need. If Namita had constructed her research prompt around the assumed-required custom logic, the language model would have produced detailed guidance on building that custom logic. The research would have looked productive but it would have been confirming an unnecessary architectural decision rather than testing whether the architectural decision was correct in the first place.

## Constructing the Prompt Live

Colin then walks through the construction of the actual research prompt in real time, narrating each addition and the reasoning behind it. He starts with a deliberately broad framing. "I want to do some deep dive research can understand built-in git CLI that can help me." This opening sentence does two things. It signals depth, which orients the language model toward thorough rather than cursory work. It also constrains the inquiry to built-in tooling, which prevents the model from drifting toward custom solutions that would reproduce the confirmation-bias failure mode.

He then layers in context about the specific work that the team has been attempting. He says, "What would Imparma do? What are they doing? Well, they're trying to use basically effectively rejects matching. They're not really doing this in a different way." The transcription renders Anupma's name as "Imparma" but the context makes clear that Colin is referring to Anupma's existing approach. The purpose of including this information is not to constrain the model to that approach but to give it a concrete starting point that it can either validate or replace. The model now knows what is currently being attempted and can search for either canonical alternatives or improvements to that approach.

Next he adds a structural requirement. "I want to persist the relationships, chronology, and full metadata. This is always a good thing." This is a critical addition because it specifies attributes of the desired output without specifying the output itself. The research must surface ways to capture relationships, ways to capture chronology, and ways to capture full metadata. Whatever the language model comes back with must satisfy those attributes. The model is free to propose any architecture that meets those requirements, which preserves the open-ended quality that Colin had emphasized as essential to good research.

## The SQLite Trick

Colin then introduces a prompt-engineering technique that is one of the most operationally useful elements of the entire teaching block. He explains, "Pretty much anything you're doing, if you bring SQLite into the picture, language models will think about it differently. Because they're saying, ah, you're not just trying to run a script and get a small thing out. You're trying to build a system. and databases come in because of infrastructure."

The significance of this trick is worth dwelling on. Mentioning SQLite in the prompt does not mean that SQLite necessarily has to be the final implementation choice. It serves as a framing signal that shifts the language model's internal model of the task. Without SQLite in the prompt, the model frames the task as a small scripting exercise and produces output appropriate for that framing. With SQLite in the prompt, the model frames the task as a system-building exercise and produces output that includes data modeling, persistence patterns, query design, and integration considerations. The same underlying research request produces materially different output depending on which framing the model adopts.

This is an example of what Colin calls "a little baby prompt engineering." He acknowledges that none of these techniques individually represents a major shift, but the cumulative effect of applying them consistently is dramatic. He says, "All research to be a little baby prompt engineering here. Nothing that's actually going to make too big of a difference, if I'm being honest. The trick to research is also specifying the format that you want it to appear in." The downplaying of the techniques' individual significance is itself a pedagogical move. Colin is teaching Namita to internalize these patterns as habits rather than as special techniques to be applied in special circumstances.

## Hypothesis as Direction, Not Answer

Colin then articulates the framing that ties together the attribute-specification principle with the prompt construction. He says, "I gave this, and I didn't say, here's what I want the solution to be. I said, here's what I think I want the solution to be." The shift from "here's what I want" to "here's what I think I want" is the entire methodology in miniature. The researcher has a hypothesis, but the hypothesis functions as a direction of inquiry rather than as an answer to be defended.

He then states his hypothesis. "So I want to see how I can go tracing, so I can see what PRs depend on other PRs, what commits depend on other commits. That can be seen as almost a timeline." This is a substantive hypothesis. It is specific enough to give the language model meaningful direction. It is general enough to allow the model to surface alternative framings, alternative tooling, and alternative architectural patterns. If the model finds that tracking pull-request-to-pull-request dependencies is the wrong abstraction, it has the latitude to say so. If the model finds that commit-to-commit tracking is the right abstraction but timeline visualization is not the right output format, it has the latitude to say that too.

## Always Include the Current Date

Colin then introduces another habit that he frames as obvious but rarely practiced. He says, "Here's another part that's really important for research. Now, this is obvious. You would think that this is completely obvious for language models. It's not. They don't do this unless you tell them to. And if they do, it's kind of by accident." He continues, "My reason for this is not because I don't think Claude can say the current date. My reason for this is Why is this important? Because these things change and you want it to be available."

The practical reason for including the current date is that GitHub features evolve frequently. A research prompt without a date anchor will produce output that mixes guidance from different time periods, some of which may be obsolete. With the date in the prompt, the model is more likely to surface time-aware analysis that distinguishes current capabilities from historical ones and that explicitly identifies recent changes. For tooling like GitHub, where major features ship continuously and command-line capabilities expand over time, the time-aware framing is materially more valuable than the time-agnostic framing.

## Do Not Over-Specify Platforms or Versions

Colin then teaches the inverse of the precision principle. While the date should be specified, the platform and version should generally not be. He says, "You'll notice that I'm not specifying GitHub Enterprise or not. Does not matter because the thing is that typically features like this, there is some analog for it. Even if it's not specifically that command that works, there will be something that's similar that came earlier." He continues, "It's very rare for things like GitHub. That they introduced some sweeping new feature that had no prior comparison."

The reasoning here is that platforms in mature ecosystems rarely introduce truly novel capabilities without some predecessor or equivalent. By leaving the platform variant unspecified, the research prompt invites the model to surface the canonical pattern rather than the variant-specific implementation. If the canonical pattern exists in only one variant, the model will say so. If the canonical pattern exists in multiple variants with different syntax, the model will surface all of them. The researcher loses nothing by leaving the variant unspecified, and gains the possibility of discovering more general or more portable approaches than the variant-specific search would produce.

## The Bazel Command Analogy

To reinforce the over-specification point, Colin uses an analogy from a different ecosystem. He says, "Usually they're just bundling things, so they're kind of like that Bazel command, you know, that can list on the dependency map. Anything that they can do, you can do. Always perceive it like that." He continues, "Anything that you can do through, or they could do like with that single Bazel command, technically if that command didn't exist, we could still do it, because what is that command doing? It's calling a series of functions that operate in the code. That's it."

The Bazel analogy makes a deeper philosophical point about tooling. A command-line tool is a convenience layer over a set of underlying operations. The convenience matters, but the underlying operations are always available to be performed directly. If a researcher accepts the convenience layer as a constraint on what is possible, the research will be artificially limited to what the convenience layer happens to expose. If the researcher recognizes the convenience layer as one of many possible expressions of the underlying operations, the research surface expands significantly. For the GitHub dependency graph problem specifically, this means that even if no single command exposes the desired capability, the underlying git operations and GitHub API operations can be composed to produce the desired capability.

## Claude Desktop versus Claude Code

The pivotal moment of the teaching block comes when Colin transitions from Claude Desktop to Claude Code. He says, "The problem with Cloud Desktop is that Cloud Desktop is like a watered down version of something that we know of that's way more powerful. And that thing, quite simple, is just Cloud Code. That's it. That's all." The transcription consistently renders Claude as Cloud, but the meaning is unambiguous.

He then takes the same prompt, copies it, and pastes it into Claude Code with two new rules added. He says, "I gave it two new rules. I said I want all the work to be organized neatly into folders and files, put an image in dimensions. I gave it another thing, make a rules.md file as you go, to keep track of my rules." The folder-and-file organization rule converts the research into a navigable directory structure rather than a single document. The rules.md file rule creates a self-documenting record of the operating principles that the agent is following, which makes the research reproducible and the methodology transferable.

Colin then explains why Claude Code is materially more powerful than Claude Desktop for research work. He says, "Whenever we go and do this, what's the big difference? Well, what happens when you run out of context in the cloud desktop? Well, you try to start a new cloud session, and you try to say, hey, pick up where we left off. And it doesn't include what you're talking about. But if you treat it almost like it's a code base, you see where I'm going with this? It gets a lot better. Because just like Claude can handle giant code bases, well, research files are really simple. Research files are just a bunch of markdown files. There's nothing else there."

The codebase-style persistence model is the conceptual breakthrough. Research is just files. Files persist across sessions. New sessions can read the files and pick up where prior sessions left off. The context window is no longer a constraint on the depth or duration of the research because the research is not living in the context window. It is living on disk. The context window only needs to hold enough information for the current operation, not the entire history of the work. This single shift changes what is possible in terms of research scope, depth, and continuity.

## Parallel General-Purpose Agents

Colin then introduces what he calls "one of the most important rules that we can have." He says, "When you are doing you to do each wave with parallel and on purpose agents in the right directions." The transcription is rough here but the meaning is clear in the context that follows. He continues, "Another difference with cloud code versus desktop is that cloud code is allowed to use multiple threads. Cloud desktop gets one. So what that means is that whenever we're doing research, we can do it at a much more rapid pace. It doesn't save some tokens, trust me on that. But we can do this at a much more rapid pace because we're able to do it in parallel."

The parallelism point is operationally significant. A research task that would take a single agent thirty minutes to execute sequentially can be decomposed into parallel sub-tasks that complete in five to ten minutes total. The token cost is higher in absolute terms, but the calendar time is much lower. For an engagement where the daily update cadence is the operating norm, the calendar time is the binding constraint, not the token cost. Colin is being honest that the parallel approach does not save tokens. He is also being clear that token savings are not the goal. The goal is rapid, deep, well-organized research output, and parallelism is the mechanism for achieving it within practical time windows.

## Never Use Explore Agents

Colin then states what is perhaps the single most important operational rule of the entire teaching block. He says, "Parallel general purpose agents. Give this one in your mind and just memorize it. Never ever let Claude use explore agents. There's one very good reason. Explore agents are unable to write files." He continues, "Which means that it's almost like it's playing the telephone game. You know, just another explore agent, explore agent will come back. Oh, I've got this great rich context. It's beautiful. And then what happens? Claude summarizes it and you lose all that rich detail."

This rule deserves emphasis because it inverts what most users assume about specialized agents. The intuition is that an agent named Explore is the right choice for exploratory research. The reality, as Colin teaches it, is that Explore agents lose information at the boundary between the agent and the main session. Whatever rich context the Explore agent gathers must be compressed into a summary to be returned to the main session. The summary loses the texture, the nuance, and the specific details that make the research valuable. The general-purpose agent, by contrast, can write its findings directly to files. The files preserve the full unabridged content. The main session can then read those files at full fidelity rather than receiving a lossy summary.

Colin frames this in terms of return on investment. He says, "You want the research agents to be worth all the tokens that they're consuming. So, you don't want the collision notes here. You want this to be the full, unabridged, deep dive." The token cost of a parallel agent is meaningful. The whole point of incurring that cost is to obtain rich detailed output. Using an agent type that loses the rich detail at the boundary defeats the purpose of incurring the cost in the first place. The rule is therefore not just about output quality but about cost efficiency. Parallel general-purpose agents that write to files are the only configuration that justifies the token expenditure with corresponding research value.

## Permissions and Validation

Colin then introduces the operational discipline that surrounds the parallel agent execution. He says, "This next part is very important too. With full web access and rate permissions. To start it, you must prove we can have these agents with these permissions. You never to do work in our main session. The agents must be proven to work before continuing. If any permissions are Json or needed, look about Json, which controls Cloud's permissions, by the way. Please take action to resolve, do not proceed, if not in this way." The transcription renders write permissions as "rate permissions" and references settings.json as "Json" but the meaning resolves cleanly in context.

The pattern is clear. The agents must have full web access to perform meaningful research. The agents must have write permissions to persist their findings to files. These permissions must be validated before the main session begins consuming tokens on substantive work. If the permissions are insufficient, the operator must adjust settings.json to grant the necessary permissions before proceeding. The validation step prevents a common failure mode in which the main session begins work, dispatches parallel agents, and then discovers that the agents cannot write files or cannot access the web. By that point significant tokens have been spent and the work has to be redone.

## Single Agent First to Confirm Query

The final operational rule that Colin states is a recommendation for query validation. He says, "I'll say I recommend doing a single agent web search, and be proud of write-up findings to confirm a query with an underscore." The transcription is rough but the pattern is clear. Before committing to a parallel multi-agent execution, run a single agent against the query to confirm that the query is well-formed and that the resulting research direction is productive. The single-agent run is cheap in comparison to the parallel run. If the query needs adjustment, that adjustment can happen before the larger token expenditure begins.

## The Full Pattern Summarized

Pulling together everything Colin teaches in this segment, the methodology has a clear shape. The researcher begins by defining the attributes of the desired outcome rather than the outcome itself. The prompt mentions SQLite or some equivalent persistence framing to trigger system-level thinking from the language model. The prompt always includes the current date. The prompt does not over-specify platforms or versions, allowing the language model to surface canonical patterns rather than variant-specific implementations. The prompt specifies the output format, with markdown files in organized folders being the preferred form for research work.

The execution happens in Claude Code rather than Claude Desktop because Claude Code supports the codebase-style persistence model and the parallel agent execution model. The researcher writes a rules.md file as part of the work to preserve the operating principles. The execution uses parallel general-purpose agents, never Explore agents, because general-purpose agents can write to files and Explore agents cannot. The agents must have full web access and write permissions, and these permissions must be validated before the main session consumes tokens on substantive work. A single agent run validates the query before the parallel scale-up. The output is a directory of markdown files that can be referenced, extended, and shared as a persistent research artifact.
