## 4. Knowledge Graph Versus Dependency Graph Terminology

The second major technical thread was opened by Namita, who had spent the week working with Justin on the build-artifact pipeline and the dependency relationships inside individual builds. Namita clarified what the team now understood after the Wednesday conversation with Justin. There are two distinct artifacts that had been conflated in earlier discussion: a dependency graph, and a call graph.

The dependency graph, as Cisco uses the term, is produced by a Bazel command that Justin shared. Given a particular build, the command returns the chain of image dependencies that go into that build. Namita gave the example of the Core 64 and N9000 Bazel packages: to build N9000, the command reveals that image one depends on image two, image three depends on image four, and so on up the chain. The output is scoped to a single build and describes which sub-images are required to produce the final package. It is executed on the Aurora Development Server. Critically, Namita confirmed when Colin pressed that the command is not custom Cisco tooling. "I think it just took a command, because when I searched it online, they just have normal commands in Bazel, which we can we can just execute." It is a built-in Bazel dependency query.

The call graph, as Cisco uses the term, is a different artifact, and it already has an owner on the Cisco side. Namita relayed that Justin had messaged the evening before the standup to say that he had created some call graph content. In the call graph framing, the relationships are function-to-function within a program rather than image-to-image within a build. "Yeah, within a program, how the functions call each other, kind of that information."

Colin surfaced a terminology mismatch for the team's internal awareness. In conventional software engineering usage, what Cisco calls a dependency graph would more naturally be called a call graph, because it describes how a build artifact reaches its own inputs. What Cisco calls a call graph would more conventionally be described as a relationship map or a function call graph within a single binary. Colin flagged this for the team's benefit but said he would not fight the terminology battle with Srinivas. "I won't die on that hill of terminology, but whatever." The practical value of the terminology note is that when the team reads external documentation or Bazel community material, they need to mentally translate between the two vocabularies.

Colin also captured the strategic implication of the Bazel-built-in finding. On Wednesday, Devakar had framed the dependency graph work as something Cisco had built. Namita's confirmation that it is a Bazel command shifts that framing. "In a way, it kind of discredits, right? And in a positive way, it discredits what Devakar was saying, like, oh, we built this. You didn't build anything. You ran a script. And by ran a script, you ran a single line in a terminal." This repositions the conversation in BayOne's favor because it opens room for BayOne to propose genuine value-add layers above what comes out of Bazel without needing to displace a claimed custom system.

## 5. Commit Attribution Script

The third technical thread was the commit attribution work Namita had been writing. The objective is to answer a specific forensic question: given a build that failed and given that multiple commits were rolled into that build, which commit actually caused the failure. Namita described the input-output contract concretely. If build number 100 includes five commits, and the build fails, the script ingests the build log files and the commit metadata directory that Justin identified, and it returns the specific commit responsible, for example commit number three. The script is working against the temporary ADS where Namita has access.

Namita was clear that this first step has been handled and that the natural next step is to extend the attribution outward: given commit number three caused the failure, which prior commits did it depend on, and could any of those dependencies be the true root cause. That second step is where the dependency graph and the PR-to-PR mapping questions intersect. As Namita put it: "there might be some previous commits, some dependent comments. which may have caused that problem. So I'm trying to figure out that link."

When Namita asked Justin whether Cisco already had a pull-request-level dependency graph analogous to the build-level one, the answer was that no such artifact exists. Cisco only has the Bazel build dependency graph. PR-to-PR dependency relationships are not currently captured.

## 6. Software Bill of Materials and the Monday Link

Colin's Monday link into the GitHub Software Bill of Materials was the fourth technical thread, and it surfaced as a pointed accountability moment. Colin had sent a link on Monday (Namita clarified during the discussion that it was Wednesday rather than Monday, although Colin's framing remained that he had flagged the direction early in the week) pointing to how the Software Bill of Materials and GitHub dependency APIs can trace PR-level dependencies even outside the NX repository itself.

Namita summarized the knowledge she had extracted from the link. If a build fails because a third-party package broke, the Software Bill of Materials query is the tracing path. It identifies which pull request updated the third-party package that failed. If the failure is instead caused by internal Cisco code rather than a third-party package, the call graph query is the tracing path. The two queries are complementary: Software Bill of Materials handles the external dependency surface, call graph handles the internal code surface.

Colin's pushback was that Namita had the knowledge but had not operationalized it. The commit attribution script was still limited to log parsing against the directory of commits, and it had not been extended to consume Software Bill of Materials data. Namita countered that the build directory available to the team does not include GitHub information, so the Software Bill of Materials path had not seemed directly relevant. Colin's rebuttal was the strategic intent behind the original link: "the reason why I brought that up on Monday was because that was the way for you to get that information. If you have access to the repository at all in any way, even if it's not that repository, even if it's another Git repository, it still stands. Because GitHub PRs and commits are treated the same regardless of what repo they're in. So even if it was a personal repo with some code, there's still a way to trace commits and PRs together. and build that map and run this." The research was not about reading the link to summarize it; it was about using the technique the link described on whatever repository access the team actually has, building the map, and running the trace.

**Internal note:** This exchange was the most direct accountability moment of the meeting. Colin's framing was that the team had used the Monday link as reading material rather than as a technical direction to execute, and that if the AI team is not using AI tools aggressively to accelerate this kind of research, it undermines the team's charter. "We're the AI team. We have to use AI more than anyone else. And we have to use it in the best way to accomplish our problems the fastest."
