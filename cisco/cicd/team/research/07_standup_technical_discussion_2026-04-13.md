## 1. WebEx Developer Platform Taxonomy (Saurav's Architectural Breakdown)

While surveying the Cisco WebEx developer setup in preparation for continuing work with Naga and Justin, Saurav identified the full taxonomy of application surfaces available inside the WebEx developer portal. In his words, WebEx exposes "integrations, service apps, bots, agentic apps, and OAuth flows." Each of these represents a distinct deployment and permission model, and the choice between them has significant downstream implications for the Pulse and Scribbler projects that Naga and Justin are leading.

Saurav's proposed architecture, which he framed as a better structural alternative to whatever the Cisco team has loosely assembled so far, is explicitly decoupled. He described the preferred shape as follows: a service-level application scrapes data from WebEx into a database, an MCP server sits on top of that database, and any bot, agentic app, or user-facing application sits on top of the MCP. As he put it, "it would be good to have like a service level app which can like read and scrape the data and put them in a DB, then create an MCP of that DB, and then create an app or a bot so that everything is decoupled."

The decisive architectural concern in this proposal is authorization and data isolation. Saurav flagged a specific failure mode if the team defaults to the naive bot-token pattern. A bot token grants the bot access to every space and chat that the bot has been invited to, which means any query flowing through that bot returns data across the full union of those spaces. In Saurav's framing: "whenever the bot is accessing data from the table, we can use OAuth of the user rather than the token of the bot because bot have access to all the chats and spaces and we do not want that data to be leaked in different other chats on which he does not have access." The MCP-over-database pattern combined with OAuth-scoped queries solves this because permissions are enforced at query time against the requesting user's identity rather than against a shared service principal.

Saurav also raised the duplication concern: before building this structure, the team needs to confirm with Naga and Justin what has already been built so that BayOne does not "get a bad start" by rebuilding something that exists. Colin's posture on this was that verbal assurances from the Cisco side should not be taken as conclusive. He wants to see the code and see the architecture regardless of what the Cisco team says is already in place.

## 3. Podman Versus Docker at Cisco

The Cisco standard for containerization is Podman rather than Docker. Colin's read on the reason: "they just use Podman because it's FOSS. So I think they're worried probably about licensing for Docker." Podman is free and open source, Docker Desktop has commercial licensing attached for corporate users, and Cisco procurement has apparently decided the licensing cost is not worth paying when a functionally equivalent FOSS alternative exists. Saurav confirmed this indirectly: Podman is available from the Cisco AppStore without requiring an access request, whereas Docker Desktop requires a formal access request, which Colin attributed to Docker Desktop's admin-level permission requirements.

Saurav has already run Podman successfully in production for his bot work. Specifically, he stood up a PostgreSQL container using the PG17 base image pulled from the AWS public container registry, which does not require Docker Hub authentication. The command surface is essentially identical between the two tools. As Saurav described it: "the whole structure of Podman as well as the command is totally copy of Docker. So you use Docker PS, so you can use Podman PS. Similarly, Docker compose, so there is Podman compose and all of the commands are the same." The only real friction he encountered was changing the initial directory from which the Postgres base image gets downloaded, which he resolved by pulling from the AWS public registry rather than from Docker Hub.

Colin's recommendation to the team was to get Docker Desktop as well, specifically for proof of concept work where BayOne needs to unblock itself locally without waiting on Cisco IT to provision infrastructure. The framing to Srinivas should be diplomatic: "take some care to preface him with, hey, we're using this for local dev so we can keep moving." Colin called out database access specifically as an area where this approach pays off, because Cisco database access will be slow to get provisioned, and Docker lets the team "spin up just about anything under the sun" locally in the meantime. Colin also noted that getting familiarity with Podman is valuable regardless, because Podman is what the production ADS deployment uses.

## 4. ADS Machine Internals (Colin's Tutorial)

Saurav asked a pointed question: "I have not worked with ADS in the past. Do we need to provision different ADS for different people or one ADS in the whole team is okay?" This opened into a substantial tutorial from Colin about what ADS actually is and how it behaves.

Namita provided the expansion of the acronym, which had not been previously clear to the team: ADS stands for Aurora Development Server. It is specifically not Active Directory Server, which was a possible misreading given the acronym collision.

Colin's characterization of the ADS architecture was that each ADS machine behaves effectively as a virtual machine, likely hosted on a Proxmox-class hypervisor that manages subdivided VM slices across Cisco's larger compute pool. His explanation framed it as follows: when you have a very large physical server, for example one with terabytes of RAM, you typically subdivide it into virtual machine slices using a tool like Proxmox, and each slice is allocated some compute, some network, and some CPU. The ADS is Cisco's internal name for this pattern, which is why public documentation on "ADS" does not exist.

The operating system question turned out to be material. Colin explained that because ADS is Linux, concurrent multi-user access works by default. His exact framing: "because it's a Linux container, by default, there is possible concurrent access. So if it's a Windows ADS machine, that is going to be what says only one person at a time can access." He predicted the specific Linux distribution was likely RHEL or Debian, which Saurav corroborated with the observation that Cisco has Red Hat licensing in place. The transcript earlier flagged this as "REL 8," which is Red Hat Enterprise Linux 8.

Colin drew the analogy to Docker to make the image-construction model concrete: each ADS machine comes pre-built with certain packages installed, analogous to a Dockerfile specifying a base image and additional layers. This makes provisioning an ADS for a new user essentially a matter of requesting a slice that matches the standard package manifest.

On the parallelism question Saurav originally asked, Namita relayed what Justin had told her directly: the team only needs to request a single ADS machine, identified as "ADS some number," and everyone on the team can access that one machine simultaneously without needing to take turns. Her exact phrasing: "even I can access simultaneously, you can access simultaneously. So that should not be a problem. We should be able to work parallel on that machine." Colin affirmed this as the correct answer given the Linux OS, and used it to reinforce the broader observation that Cisco has significant tribal knowledge, meaning "things that aren't written down that should be," and that figuring out the right stack to request is consequently dependent on asking the right person.

## 6. Scribbler Versus WebEx A/B Testing Three-Tier Framework

This was the most substantial technical tutorial in the meeting. Srikar's preceding summary of what Srinivas had said on the April 10 call was that the two transcription paths, WebEx native transcription and Scribbler (which wraps Whisper), could be pursued in parallel, because both are independently useful. Saurav reframed this as an A/B test: "we are also doing with WebEx transcripts and we are also doing with this, Whisper plus this. Let's see who gets a better summary and better use case." Colin accepted the A/B framing and then insisted that the team must widen the evaluation beyond raw transcription quality.

The resulting three-tier evaluation framework Colin laid out for Saurav and Srikar:

**Tier 1 is raw out-of-box transcription quality.** This is a pure head-to-head on transcript accuracy between vanilla Whisper as it ships and vanilla Cisco WebEx transcription as it ships. The analytical depth Colin wanted here was not just "which is better" but "what specifically is better." His hypothesis, based on prior experience with Whisper: "typically with Whisper, from my experience with it, it's simply a matter of being better at picking out words," and Saurav expanded this to "a lot of corporate jargons has been trained on that, so if you are using those, it can pick those up." The expected delta is concentrated in jargon, names, and technical vocabulary rather than general prose.

**Tier 2 is glossary-augmented quality.** This tier asks whether a simple prompt-engineering layer can close the accuracy gap on whichever option is cheaper. Critically, Colin specified that this is not about preprocessing the transcript. It is about providing meta-information at the point the transcript is consumed by a downstream AI. The context that gets injected: the meeting attendees list (because "people's names are not well covered by any transcription model"), a small table of commonly missed terms with correct spellings, and an instruction to "use common sense and good judgment" when interpreting garbled phonetics. The underlying premise is that downstream AI models are fairly tolerant of transcript errors, so a 10% raw accuracy delta may evaporate entirely once both transcripts are handed to an LLM for summarization or extraction.

**Tier 3 is compute and scale cost.** This is where Colin spent the most time. The central question: if Whisper is self-hosted, which is the current Scribbler pattern, the hosting model does not multi-thread. "Generally speaking, when people use Whisper Local, it's not multi-thread. So it's not like they can do 500 meeting transcripts at a time." That means the per-request cost structure must be evaluated at the per-concurrent-meeting level, and the aggregate cost must be evaluated against Cisco-wide meeting volume. Colin used the explicit hypothetical: "if I told you that with Whisper, I could get 10% better transcripts, and it costs $20,000 a month. Are you going versus free for the other one with WebEx transcription? Are you going to say Whisper is important? No. You're going to say, that's great, but 10% doesn't justify $24,000 a year or $240,000 a year." (The numbers in Colin's hypothetical are internally inconsistent, $20,000 a month is $240,000 a year, not $24,000 a year, but the rhetorical point is the scale-multiplier effect.)

The ultimate question that sits above all three tiers: does the accuracy improvement actually matter for the downstream AI use case? Colin's answer is probably not. "Does AI afterwards care about it? If we're trying to summarize that transcript, does it matter that it's 10% more accurate because it could be completely fine. It's almost like when you mistype into Claude Code, Codex, or Cursor, you mistype something, AI doesn't care, AI just rolls along and it's fine." The thesis is that LLM tolerance of transcript errors is high enough that Whisper's accuracy advantage does not translate into measurable downstream value.

Colin also flagged the duplication concern that applies at scale. If Srikar builds an app that transcribes WebEx meetings using Whisper, and Saurav builds a different app that also transcribes the same WebEx meetings using Whisper, both apps are processing the same audio through the same model for the same purpose, which is wasteful. "You're processing the same information twice in the same way for the same purpose." This is part of why the service-layer pattern described in section 1 matters, there should be one canonical transcription service at the organization level, not a transcription call inside every individual application.

## 7. Build Log Data Modeling (Colin's Framework for Namita)

Namita's context for this discussion: she is the primary owner of the build-log analysis track, she has GitHub repo access and ADS access, and she is the only team member who has actually seen real Bazel logs (in the dev environment, from the past four days, all passing). Saurav floated a question about whether to capture user-level or commit-level information alongside the logs themselves, because Srinivas had indicated Cisco wants commit-level transparency. This prompted Colin's data-modeling framework.

**Use a star schema, not NoSQL JSON.** Colin was explicit: "this is one where realistically it should be a star schema. And I mean, that's the easiest possible way also to enable downstream agentic AI. I would not do it as like a JSON format with NoSQL. I would not do that for this one." The reasoning is that an agentic AI tool layer wants tabular, relational data it can query through a small, well-defined set of MCP tools. Star schema gives you that. Document-oriented NoSQL gives the AI an unbounded query surface that is harder to wrap in tools.

**Key on SHA hashes.** Colin called out the Git SHA hash as the canonical identifier for both commits and PRs. The SHA hash is unique and is directly queryable from Git itself, which means the table only needs to store the hash, not the full commit content. As he put it: "once you have that, you pretty much can query that information real time from Git, which is ideal. That way you don't have to worry about storing it unless it makes sense to store."

**Do not persist everything.** This is the counterintuitive part. Because Git is already a queryable system and because MCP lets the BayOne team wrap Git CLI or Git MCP as a tool, most Git-retrievable data does not belong in the warehouse at all. Colin's example: "if I wanted to see get what commits were part of a PR, do I really need to do a complicated structure to store that? No. As long as I know what the PR was and what the hash was for the specific PR push, then I can tell you from a simple reproducible Git query, what the command would be to pull that. And then we can just build a tool for MCP that can go and retrieve that on the fly." Saurav confirmed this by noting Git already exposes both a CLI and an MCP server.

**What does belong in the store:** IDs, SHA hashes, metadata (author, timestamp, build ID, log file provenance, which ADS machine produced it, which user triggered it, whether it was a user build or a nightly CD build), and derived analytics computed on top of the logs.

**Backward traceability is the core use case.** The data model must support tracing from a build outcome back to the contributing PRs and commits. This is the "commit-level transparency" Srinivas asked for, and it is enabled by the star schema plus SHA-hash-keyed joins, not by persisting full commit trees.

Namita signaled she understood the direction ("Okay, Colin") but did not push back with detailed questions in this meeting. Colin flagged that the two of them will meet separately to go deeper on the architecture.

## 8. Log File Structure Unknowns (Observation Period Rationale)

Colin was explicit that the team cannot design the log processing pipeline until it knows what the logs actually look like. The specific unknowns he called out:

**Composite versus individual log files.** It is not yet known whether the logs are a single large text file into which multiple applications append (a composite log), or a set of individual files per application or per run. Colin: "the log file could be a giant text file that is blending together logs that are from multiple different applications. We don't know whether or not that's true yet until we actually get eyes on it and see." The answer materially changes the parsing strategy, because a composite log requires source-attribution parsing as a first step, whereas individual files can be dispatched to different parsers by filename or path.

**What log types exist beyond Bazel.** Namita has visibility into Bazel CI logs from the dev environment because that is what Justin showed her. Whether there are also Jenkins logs, DevX logs, separate build logs, or other log streams is unknown. Colin: "as of right now, I only personally have awareness that there is a Bazel log. I don't know if there are Jenkins logs, DevX logs, build logs, etc." [unclear in transcript] whether any of these other log types actually exist at Cisco; this is a list of candidates to investigate, not a confirmed inventory.

**CI versus CD distinction.** Namita confirmed a key structural fact that had not been fully clear before this meeting: the CI process runs Bazel dev builds, and the CD process runs nightly builds. These are two separate pipelines with potentially different log shapes. Her exact framing: "the CI process that they have, this is the one that Justin showed me. So those are Bazel logs from the dev environment. But then there is another process that is a CD process wherein nightly builds happen. So that part Justin has not showed me." She has CI visibility, she does not yet have CD visibility, and she committed to pursuing CD visibility as the next step. Colin then clarified that the scope narrowing from Srinivas (Bazel only, Gmake excluded) applies to the CI side; the CD scope was not separately clarified on the Friday call.

**Plan: observation period.** Rather than asking Justin for a single sample log file, the team will capture logs over multiple days to build a library of known log shapes. Colin's framing: "anytime you have someone who's not an AI person give you a sample of data, you can 100% guarantee it's not going to be a good sample." He explicitly wants sampling across success and failure runs (Saurav added this refinement), across builds from different triggers, and across time. If the logs are already getting saved to a database or to an NFS file server (which Justin had indicated), BayOne should get read-only access to that NFS mount and pull logs programmatically rather than manually. ## 9. Naga's Scribbler Architecture Assessment

This was the most consequential architectural revelation of the meeting. Saurav initially asked whether the team could get last Friday's call transcript from Scribbler to compare against the WebEx-side transcript. He thought Justin had been running Scribbler for the call. Srikar corrected this: "this is from Naga. They haven't created configuration to any other meetings as such. They just have to have some code in the repo and you can run that on your local machine anytime with any audio file." In other words, Scribbler did not automatically process the Friday call. Scribbler cannot automatically process any call. Scribbler is not deployed to any Cisco system.

Saurav confirmed the deeper finding: there is no API key or service endpoint published anywhere that would let BayOne point its own audio at Scribbler and get a transcript back. There is no service.

Colin's synthesis of what Scribbler actually is, based on this information: "if this Scribbler tool is literally just essentially a Python script that runs Whisper, I mean, that's probably 50 lines of code at most. Provided a text file on the local system, use Whisper, run the Whisper model locally, and process a transcript, save a file, probably 50 lines of Python."

The implications Colin drew out:

**Scribbler is a POC at best, not a production system.** It runs locally, manually, against a single audio file at a time, produced by a human operator who has to first acquire the audio file out of band. There is no observer loop, no scheduling, no integration with WebEx, no persistence, no service layer.

**BayOne can propose a proper service-layer architecture without stepping on Naga's toes.** This is the important strategic conclusion. If Scribbler were a real service, proposing to replace it would be politically costly. Because Scribbler is effectively nothing, proposing to build a real service is not a replacement, it is the first actual build. Colin: "if it's not fully matured yet, that gives us a way to put ourselves inside of it and insert and say, we're going to do this a better way, a different way, without really ruffling feathers, because you can't get your feathers ruffled if it's not really working for real yet."

**Substantial work remains to get to a real service.** Specifically, a production Whisper-based transcription service requires an observer backend that finds new WebEx meetings, pulls the audio, pipes the audio to Whisper, captures the resulting transcript, and persists it. None of that has been built. Colin: "if you had Whisper running, you essentially now need a whole observer backend architecture, something that goes and finds new WebEx meetings, gets the audio from it, pipes it in. That work has not been done, it sounds like." This is a concrete scope of future work BayOne can define and own.

## 10. Cisco's "Solution Without a Use Case" Pattern (Colin's Critique)

Colin used the Scribbler finding to articulate a broader pattern he sees across the Scribbler and Pulse WebEx tooling: these projects appear to have been built without a clear business use case.

His framing, which he acknowledged was the mean version: "effectively what Cisco has done is they've created a solution without an actual use case. Which is another way of saying I messed around at work and got paid to do it. There's no business value to it, period."

The expanded critique: for Scribbler specifically, what problem does a locally-run Python script that transcribes a single audio file solve that is not solved more cleanly by the native WebEx transcription that already runs automatically for every meeting? For Pulse, what is the downstream consumer of the scraped WebEx data? "They're kind of just sounding like they fed it to ChatGPT and said, what could this do? Give me some ideas for this." The projects feel ideation-driven rather than problem-driven.

Colin then pivoted to how to communicate this to Cisco without creating friction. The rule: diplomatic presentation, architecture-led.

"We have to be very diplomatic. Capture that. Say, here's what we can see for the current state without saying that this is complete junk. Here's why this won't work. Here's what our proposal is to do this better at scale." The explicit instruction was: do not verbally call out the current work as bad. Show the current-state architecture diagram. Show the proposed future-state architecture diagram. Let the comparison speak. "Simply show it as a diagram and say, here's how it works today, here's what we suggest it works in the future, because it will naturally come out as to what the shortcomings are to any logical person that's technical."

As a meta-note on engineering culture, Colin also shared an observation he has built up over many consulting engagements: "whatever they tell you verbally will be about maybe 70% correct. And then 30%, they will either have misremembered, misstated, or misrepresented per the database, not intentionally, but something that is in their eyes works this way, does it actually work that way in the code?" The implication is that verbal assurances from the Cisco team should be treated as hypotheses to verify against the actual codebase, not as ground truth. Saurav connected this to the current industry pattern: "when everyone is using Codex and Copilot, till the tests are passing, everything is fine." AI-assisted coding increases the rate at which verbally-claimed functionality diverges from actually-implemented functionality, because the tests pass, the developer checks in, and no one reads the code carefully.
