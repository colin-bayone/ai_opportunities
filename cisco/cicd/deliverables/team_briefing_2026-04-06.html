03 UPCOMING WORK

First Tasks Overview

The following work items were discussed during the April 2-3 sync with Srinivas's team. This section provides a high-level overview of what is coming. We will cover each of these in detail, including specific assignments and expectations, during our next team meeting.

1 WebEx Space Scraper

Our first deliverable is a plugin that downloads the content of the NXOCI workflow WebEx space, which is where Cisco engineers have been posting their CI/CD pipeline questions and issues for the past six to eight months. Once we have that content, we will use AI to summarize the issues, categorize them, and rank them by priority and severity. This analysis becomes our starting point for understanding what Cisco engineers struggle with on a daily basis, and it will directly inform which problems we tackle first.

Naga, an engineer on Srinivas's team, may already have working code for a WebEx content scraper and a recording transcription tool. Colin will connect the team with Naga so we can evaluate what exists before building anything from scratch. There is also a public WebEx API that Cisco has published, which Srinivas indicated should take minimal effort to integrate with.

Srinivas was clear that this plugin must be built as a reusable component, not a one-off script. It will be used across other WebEx spaces and other DeepSight applications beyond CI/CD. The orchestration should run on Cisco's existing Airflow instance rather than a standalone deployment, so that we avoid creating parallel infrastructure that would need to be maintained separately.

2 Meeting Recording Transcriber

The second plugin follows the same pattern as the scraper. Given a WebEx meeting recording, this tool will download the recording, extract the transcript, produce a summary, and pull out action items. The output should be driven by a configurable prompt so that different teams can tailor the summarization to their own needs without modifying the underlying code.

As with the scraper, this must be designed for reuse across multiple contexts. The same reusability and Airflow orchestration requirements apply.

3 Build Log Analysis with Justin Joseph

The team will meet with Justin Joseph for one to two working sessions to develop a thorough understanding of the build log structure at Cisco. There are two distinct build types that we need to understand: official builds, which run nightly and accumulate commits from over 1,000 engineers into a single build, and user builds, which are triggered by an individual engineer submitting a single pull request. Both produce logs in the same format but at very different scales.

During these sessions, we need to understand the log format, the typical log size for each build type, the common failure categories, and what tools Justin's team has already built. Justin already has some AI-based fix suggestion tooling in place, but there is currently no capability to determine which specific commit caused a build failure. That gap is one of the problems we will be solving.

Srinivas was explicit that this understanding must come before any design work on failure analysis. The team should treat these sessions as a prerequisite, not something to rush through in order to start building sooner.

04 DESIGN CONSTRAINTS

Rules for All Work

These constraints apply to everything we build on this engagement. They come directly from Srinivas and are non-negotiable.

Build for Reuse

Everything we build will be used beyond CI/CD. Srinivas was explicit about this: do not assume anything is single-purpose. Plugins, MCPs, Airflow pipelines, and any other components must be designed so they can be consumed by other teams and other DeepSight applications without modification.

Inference Cost Discipline

Cisco's inference costs have already increased by 4X, and Srinivas will reject any approach that throws large logs or raw data at a language model without proper preprocessing and engineering. Solutions must be designed to minimize token consumption through structured extraction, filtering, and targeted prompting. This is not a nice-to-have; it is a screening criterion for the work we deliver.

Understand Before Designing

The team must spend time with Justin and the build infrastructure before proposing any architecture. Srinivas wants everyone to understand the artifacts, the log structure, and the failure modes firsthand. No design work should begin until that understanding is solid and verified through direct observation.

06 CONTEXT

MCP as the Integration Pattern

All data sources feeding the CI/CD application on DeepSight need to be exposed through MCPs (Model Context Protocol). This is how our work plugs into the platform.

The workflow Srinivas defined:

  1. Inventory existing data sources and databases
  2. Check if MCPs already exist for each source
  3. If yes, evaluate the tool calls they support
  4. If no, create MCPs
  5. Map tool calls to user needs
  6. Design all tool calls to serve both self-serve (user asks a question) and agentic (system takes action) modes

08 CONTEXT

Call Graph and Impact Analysis

The CLAM/LAM system can generate call graphs at compile time, showing function-level caller/callee relationships. Two approaches:

Approach Description Use Case
Baseline Generated from nightly builds, stored as a lookup database Known code: "if this function changes, what is the impact?"
Dynamic Generated on the fly for new or modified files not yet in the baseline New additions that the nightly build has not picked up

This is distinct from CRFT, which runs on the physical switch and tracks runtime function coverage. The call graph is static analysis at compile time. Both are independent data sources.
