Open Items and Access Status

Status for the Cisco CI/CD engagement covering this week's workstreams and the access items gating the end-of-next-week deployment target. Four open items are in motion. Four access items require Cisco-side unblocking before a DeepSight deployment can take place.
Open Items
In Progress
Issue Categorization Skill and Dashboard
Advancing

Three years of NX-OS CI workflow chat have been scraped and fully categorized into 78 categories across 12 top-level groups and 66 sub-groups. The categorization runs through a deterministic keyword pass followed by a language-model pass for ambiguous messages, with full traceability back to the source message. All of this is backed by a SQLite database and surfaced through an Apache eCharts dashboard with weekly, monthly, and quarterly views, heat maps, treemaps, and click-through drill-down on every category. The skill itself has been pushed to the CI/CD repository so the categorization can be re-run as new messages arrive.

The gap today is that the dashboard is a one-time artifact. To be useful to a release manager it has to run continuously and feed live insights back into the CI/CD workflow. That requires a deployment target on the DeepSight platform, an ingestion loop for new WebEx messages, and an integration layer that maps each category to the appropriate response path (bot, MCP lookup, or human escalation).

Needs
Confirmation of the deployment target and the minimum viable form of real-time delivery for release managers so the live version can be scoped tightly.
CAT MCP Integration
Tools Identified

The CAT MCP has been installed from the DeepSight marketplace in VS Code and the four tools it exposes have been identified and documented. A gap analysis is underway between those four tools and the CAT category issues surfaced in the categorization work, so that the mapping between an issue and the right tool call is deterministic before the bot layer is wired.

End-to-end path defined: an incoming CAT-category issue in WebEx triggers a CAT MCP tool call, the response feeds an intermediate reply layer, and the WebEx bot posts the reply back into the channel. Execution of the full path is currently blocked because commit-approval-tracking access on the NX repository has not been granted; without that access the tools are visible but cannot be exercised against live data.

Needs
NX repository access so the four CAT MCP tools can be exercised end-to-end against live commit data this week.
Build Log Commit Attribution
Script Working

A script that identifies which commit caused a given build failure is working today on the temporary Aurora Development Server. It consumes the build log and the commit metadata directory for a given build, isolates the failing target, and returns the implicated commit.

The Bazel per-build dependency graph is confirmed as a built-in command rather than custom work, and is being used as the on-demand substrate for localizing failures within a build. The next step is moving from commit attribution to PR-to-PR dependency mapping, so that when a release lead decides to back out a PR, the tool returns the other PRs that must be backed out with it. That requires either a full-job Bazel dependency graph (rather than per-step) or a GitHub-native dependency trace across merged PRs in the nightly build.

Needs
Availability of a full CI/CD job dependency graph, and confirmation that GitHub-side dependency information is accessible for PR-to-PR mapping.
