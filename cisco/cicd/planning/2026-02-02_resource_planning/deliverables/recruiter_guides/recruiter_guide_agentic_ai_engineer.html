Recruiter Guide

Agentic AI Engineer

Remote (Offshore) | Autonomous Systems + Self-Healing CI/CD
Helpful for pre-screening: If your screening call is on Teams, please enable transcription and share the transcript afterward. This helps our technical team review candidates more effectively. Not required, but appreciated!
What makes this role different: "Agentic AI" means AI systems that take real actions automatically - not just answer questions. This person builds systems that detect when something fails, decide what to do about it, and fix it without human intervention. Think of it like building an AI that can push buttons, not just give advice.
Must-Have Skills
  • 4+ years software engineering, 3+ years AI
    Production experience building AI systems that take real actions, not just chat or analysis
  • Autonomous/agent system experience
    Built AI that does things automatically - retries jobs, fixes issues, takes actions based on decisions
  • LangGraph or similar agent frameworks
    Frameworks for building AI that plans, reasons, and calls tools. LangGraph preferred.
  • Tool-calling / function-calling patterns
    LLMs that invoke APIs, run commands, or modify systems - not just generate text
  • Safety patterns for autonomous systems
    Circuit breakers (stop if something goes wrong), limits, rollback, human approval for risky actions
  • Python backend development
    Systems-level code, APIs, error handling, retry logic
  • CI/CD understanding
    Knows how build pipelines work (Jenkins, Airflow). This role automates pipeline failures.
  • AI pair programming proficiency
    Actively uses Claude Code, Cursor, or similar agentic AI tools daily. Not just autocomplete (Copilot) - should be comfortable with AI that writes and edits code autonomously
Nice-to-Have Background
  • SRE or platform engineering
    Site reliability engineering - keeping systems running, incident response, automation
  • Self-healing system experience
    Built systems that automatically recover from failures without human intervention
  • Airflow or Jenkins experience
    Specific tools used by our client for pipelines and orchestration
  • Governance/policy framework design
    Defined rules for what automation is allowed to do vs what needs human approval
  • Distributed systems knowledge
    Understanding of failure modes, retries, idempotency (running something twice gives same result)
  • Monitoring and observability
    Grafana, Prometheus, or similar - tracking system health and agent behavior
Strong Signals
  • Built systems that take real actions in production (not just make recommendations)
  • SRE, platform engineering, or production operations background
  • Experience with incident response automation or auto-remediation
  • Designed governance rules for automated systems
  • Enthusiastic about AI tools - can describe specific ways Claude Code or Cursor changed their workflow
  • Understands failure modes and how to build systems that fail safely
Warning Signs
  • Only built chatbots or Q&A systems - no experience with AI that takes actions
  • Can't explain how to prevent an autonomous system from doing something harmful
  • No production experience - only demos, prototypes, or research
  • Doesn't understand retry logic or idempotency (critical for safe automation)
  • Doesn't use AI coding tools, or only uses basic autocomplete (Copilot alone)
  • Vague about what they personally built vs what their team built
  • Poor written communication (critical for remote work)
Screening Questions

Record all screening calls. These questions reveal depth without requiring technical evaluation. Note answers verbatim for technical review.

"Describe an AI system you built that takes real actions - not just answers questions. What actions did it take?"
Weak Answer
Can only describe chatbots or Q&A systems. No examples of AI that actually does things. Confuses "recommendations" with "actions."
Good Answer
Describes a specific system that took real actions - triggered deployments, restarted services, modified configs, retried jobs. Can explain what safeguards they put in place.
"How do you prevent an autonomous AI system from doing something harmful or unintended?"
Weak Answer
Hasn't thought about this. Vague answers like "test it well." No specific safety mechanisms.
Good Answer
Describes specific patterns: limits on actions, circuit breakers (stop if too many failures), human approval for high-risk actions, audit logging, rollback capabilities. Has actually implemented these.
"What's the difference between a chatbot and an agentic system?"
Weak Answer
Doesn't know, or thinks they're the same thing. Vague about what "agentic" means.
Good Answer
Clear explanation: chatbots answer questions, agentic systems take actions. Can explain planning, tool use, multi-step reasoning. Understands the safety implications of systems that do things vs just talk.
"How would you design a system that automatically retries failed CI/CD jobs safely?"
Weak Answer
No idea, or suggests unsafe approaches ("just retry everything"). Doesn't mention limits or safeguards.
Good Answer
Thinks about: which failures are safe to retry vs which need human review, limits on retry attempts, exponential backoff, logging what was retried and why, not making things worse.
"What AI coding tools do you use, and how have they changed the way you work?"
Weak Answer
Doesn't use AI tools, or only mentions basic autocomplete like Copilot. Can't describe specific ways it changed their workflow. Dismissive about AI assistance.
Good Answer
Uses Claude Code, Cursor, or similar agentic tools. Gives specific examples of how it changed their work - faster debugging, writing tests, exploring unfamiliar codebases. Enthusiastic about the productivity gains.
"Can you explain [most complex thing on their resume] to me like I'm not technical?"
Weak Answer
Uses jargon, gets frustrated, or can't simplify it.
Good Answer
Uses plain language and analogies. Checks if you're following. Patient and clear. (You can evaluate this directly.)
