## Overview

This document captures a specific coaching moment in the third Srinivas team meeting on 2026-04-17. During the discussion of the build-log analysis architecture, the conversation pivoted into the question of how an AI-applied build fix can be verified as functionally correct, not merely syntactically compilable. Srinivas surfaced the concern. Justin proposed an initial verification model centered on pull requests. Srinivas pushed back with the now-recurring "surgery is successful, patient is dead" metaphor, asserting that passing a build is insufficient evidence of correctness. Colin then contributed the BayOne AI practice's operational framework for automation decisions, stating three levers: impact, complexity, and experience. Srinivas received the framework with agreement and a recognition that Cisco and BayOne are on the same page. This is the first time in the main research chain that Colin articulates this framework to Cisco directly, and it sets the governance posture for all subsequent automation conversations in the engagement.

## Srinivas's Trigger Question

The exchange began as Justin described the in-progress triage work on build logs, noting that the tool appeared capable of fixing compile errors by inserting corrections such as additional conditions. Srinivas interrupted the description with a direct challenge, asking, "no, but how do you know the fix is valued?" He then sharpened the question: "When you say build failure, let's say there is a build failure, and you ask the agent to fix it, it added an if condition. How do you know? Build will pass, but how do you know that if condition is correct?" The concern here is fundamental. A build passing is evidence only that the code compiles or that the immediate error condition has been removed. It is not evidence that the inserted logic preserves the intended behavior of the surrounding system. A large language model can produce a compilable but semantically incorrect patch, particularly in a complex build environment like NX-OS where conditional logic may control platform-specific behavior, feature gating, or low-level hardware interaction. Srinivas's challenge forced the conversation from the mechanics of applying a fix to the epistemics of trusting a fix.

## Justin's Initial Verification Model

Justin responded with a pull-request-based verification model. His framing was that the tool would not auto-merge a change. Rather, it would propose a change through a pull request and rely on an established review step before the change is brought in. He said, "we can create a PR and use their own data review before they bring it in." He acknowledged that incorrect fixes would happen and framed this as a feedback loop for the tooling itself: "If that change is not correct, we can improve the tool for the next time... we can note that down." He closed with "we do verify." On the question of logical correctness, Justin was candid that build-pass is only one verification dimension. He said the PR cannot be considered converged on the basis of a passing build alone, noting that the team would still go through full build validation after the change lands. Translated from the transcript, the position is that syntactic and integration validation are covered by the existing pipeline, but logical correctness remains a human judgment call made during code review.

## The "Surgery is Successful, Patient is Dead" Exchange

Srinivas was not satisfied with the PR model alone and pressed the functional verification question explicitly. He said, "But functionally, how do you verify that fix is correct?... That's my point that without user in the loop we cannot auto fix." Justin then elaborated on the human-in-the-loop constraint, framing the log analyzer's job as analysis and suggestion rather than application. In his words, paraphrased from the transcript, the log analyzer should do the log analyzer part, analyze the build failure, categorize it, and then either through the integrated development environment or the developer's workspace, make a suggestion surfaced in the log analysis, but not let the user take that suggestion automatically. If the developer chooses to incorporate the fix, that becomes a subsequent commit. Justin explicitly acknowledged he was reiterating Srinivas's concern.

Srinivas then introduced the metaphor that now recurs throughout the engagement: "It's like a surgery is successful, patient-led," which the transcription rendered awkwardly but clearly meant "patient is dead." The point is that optimizing for the immediate measurable outcome (the build passes, the surgery completes) while failing on the underlying purpose (the code is correct, the patient survives) is a pathological success. As long as the build is passed, Srinivas observed, users will accept it at face value. Nobody will audit why it passed or whether the particular inserted condition created a latent defect. Building an audit system after the fact to catch these cases retroactively would be extremely difficult.

Colin later echoed the metaphor to validate Srinivas's framing and to connect it to a more general pattern in AI deployment. He said, "you know, I want to lose weight. So why do not I just lose one of my legs? Technically, I have lost weight. But like you said, the surgery of successful patients is dead. So it is the same, same thing with AI here." The shared vocabulary here is valuable. Both sides of the table now have a short phrase that captures the failure mode: the AI satisfies a narrow success criterion while violating the broader intent.

## The Suggestion-Not-Application Pattern

Srinivas resolved the tension with a clean principle. The AI proposes, the human decides. In his words, "if you keep the suggestion to the user and it is in the user hand, he is assembling. He is going to figure it out, whether he likes it, dislikes it, or you want to fix it in some other way. For us, it is just a suggestion." The word "assembling" is unclear in the transcript and may be a dictation artifact for "accountable" or a similar word, but the direction is unambiguous. The AI's output is advisory. The developer retains responsibility for accepting, rejecting, or modifying the proposal.

Srinivas then raised the follow-on mechanics question: "only thing from the input point of view, how does user say that, let's say, you suggested a fix, right? How does he apply the suggestion in his workspace?" This is the practical workflow question. Once the AI produces a suggestion, what is the physical path by which the developer reviews it, accepts it, and gets it into their code?

## The Topic-Branch Model

Justin answered with the topic-branch model that Cisco already uses for developer isolation. Each developer creates their own topic branch before their changes merge to the integration branch. The agent would create a pull request into that personal branch, not into the shared trunk. In Justin's words, "They'll have longer data branch and then whatever coming so we can create a PR into that branch so that they can see that change they can also commit to that branch if they want to change it or if they want to just commit to their branch and not accept that PR that's fine as well." This model is important for three reasons. First, AI-suggested fixes enter the developer's personal workflow rather than the main trunk, so incorrect suggestions cannot pollute shared branches. Second, the developer has the full range of responses available: accept as is, modify and commit, or close the pull request without merging. Third, the topic branch creates a natural audit boundary. Every AI-suggested change and every developer decision about it is captured in git history on a branch scoped to that developer.

## Colin's Three Automation Levers Framework

Colin delivered the framework as a structured response to the verification concern, positioning it as "the three levers for the automation that we have, that we've done." In other words, this is not an ad hoc suggestion. It is the BayOne AI practice's established operational framework for deciding how much human involvement an AI-applied change requires.

The first lever is impact. Colin said: "even if you are going to have AI apply effects, auto-approve impact is one thing. How much does this impact? Is this something that's a low severity, high severity fix? What are the downstream consequences of this being right or wrong." Impact here is about blast radius. A fix in a utility function used by three callers has less impact than a fix in a platform initialization routine used across every build.

The second lever is complexity. Colin said: "if something is very complex, that is more likely to need a human in there to verify that." Complexity is a proxy for the probability that a simple-seeming fix introduces hidden interactions or that the AI lacks sufficient context to make the right call.

The third lever is experience. Colin said: "what has happened in the past that has been able to be a past fix? So if AI is going to apply a fix, what is your confidence that it's actually going to resolve it correctly versus a human needing to be there?" Experience is historical track record. If the AI has proposed a hundred fixes of this category and ninety-nine were accepted without modification, the confidence is high. If acceptance is fifty percent, the confidence is low.

Colin added a calibration observation: "Some things it varies drastically across the board. So sometimes things are 99 percentile, sometimes things are 50 percentile for when AI comes up with an acceptable fix." The framework is not a binary gate. It is a multi-dimensional scoring that should vary by fix category, and the data to calibrate it comes from running the system and observing acceptance rates.

## Srinivas's Receptive Response

Srinivas did not deflect or challenge the framework. He received it. His responses were "I think we have been on the journey for a long time, just and all and so," "Yeah, we are trying various tasks. You'll be amazed," "Both up on the same page," and "Yeah, I agree." The tone is agreement combined with recognition that Cisco has been doing related work internally and that Colin's framing lines up with their own developing understanding. This is significant. It means the three-levers framework is now shared vocabulary for the engagement. Future automation decisions can be discussed in terms of impact, complexity, and experience, without the framework itself needing to be re-established.

## The Implicit Risk Taxonomy

Although Colin did not write out a matrix, his framework implies one. High impact combined with high complexity combined with low experience produces a strong requirement for human review. Low impact combined with low complexity combined with high experience produces conditions where AI auto-application becomes defensible. The middle cases are context-dependent and should default to human-in-the-loop until experience data accumulates. This taxonomy is useful as the engagement progresses because it gives the team a language to negotiate which fix categories can eventually move into more automated modes once the tool has proven itself on simpler, lower-impact categories.

## Connection to the Broader Engagement

The framework connects directly to the MCP endpoint design that Srinivas outlined earlier in the meeting. Srinivas asked BayOne to expose build failure analysis as a Model Context Protocol endpoint that the existing functional-fix agent can consume. For the three-levers framework to operate, that endpoint must return enough metadata for the agent, and eventually for an automation layer above the agent, to make impact, complexity, and experience judgments. This means the endpoint cannot just return a textual categorization of the error. It must return structured information about the affected components, the scope of the change being proposed, the historical track record for similar errors, and an indication of complexity. The framework also connects to the time-to-resolution metric discussed in the pain point analysis earlier in the meeting. Not all fixes require human verification. Some can be auto-applied. The measurement should therefore separate the two paths, because a single blended time-to-resolution number will obscure the fact that the auto-apply path is near-instantaneous and the human-in-the-loop path has a floor set by developer response time.

## The Single Dashboard, Single Feedback Mechanism Goal

Srinivas closed this section of the meeting with a framing that pulled together the functional-fix agent, the build-failure analysis track, and any future log-analysis surface. He said the goal is a "single dashboard, single feedback mechanism single way of interacting for his PR." From the user perspective, whether a suggestion comes from a functional correctness finding, a build failure fix, or a log analysis insight, the interaction pattern should be identical. BayOne's contribution in this engagement is to ensure that the build-failure path produces suggestions that flow through the same interaction pattern, using the same agent, the same pull-request semantics, and the same topic-branch model that the functional-fix work has already established.

## Governance Posture

The meeting reinforced several governance rules that now apply across the engagement. AI-applied fixes require a user acceptance path, and no direct-to-trunk auto-merges are acceptable. All changes flow through the CICD pipeline, which reinforces the bot deployment governance rule surfaced earlier in the meeting where Srinivas stated that no private repository deployments are allowed and that any code change must be committed and go through the process. An audit trail must exist for every suggestion, every accept or reject decision, and every resulting build outcome, because without it the team cannot calibrate the experience lever over time.

## What the Framework Does Not Cover

Several verification concerns fall outside the scope of the three-levers framework as discussed in this meeting. Runtime errors discovered after deployment are outside the build-log triage scope and were not addressed. Logical correctness verification beyond build-pass requires test suites and quality assurance processes that are not part of this engagement's scope. Security implications of AI-applied changes were not discussed in this meeting and would need separate treatment if they become relevant.
