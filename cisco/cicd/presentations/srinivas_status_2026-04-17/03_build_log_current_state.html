Infrastructure and Existing Automation

The team completed two working sessions with the build infrastructure group on April 8 and 9, verified the log landscape hands-on, and reviewed the existing build repair automation pipeline at the code level.

Build System and Log Landscape

Two build systems in use: Gmake on the legacy main repository and Bazel on the active development repository, with Bazel recommended as the primary focus
Build metadata stored in MySQL, logs stored separately on NFS via ADS machines in a flat directory structure
CI and CD builds differ: different NFS path patterns, different retention policies, and only CD builds have a build_log.html entry point
Retention disparity: CI failure logs kept 3 to 5 days, CD failure logs retained for 5 years, informing where historical analysis is feasible
Log scale: 40 or more individual log files per build, with 13 distinct Bazel error log types catalogued across images and stages

Existing Automation: 7-Component Pipeline

Pipeline reviewed at the code level (PR 642 on the DCN tools repository), confirming architecture and identifying concrete gaps
Incomplete context extraction: the mtime-based file selection and backward scan that stops at ARCH lines miss relevant information
No persistence across runs: analysis results are not preserved, so fixes cannot be traced back to root cause
Gmake inefficiency: rebuilds all targets on every retry, while the Bazel path correctly tracks remaining failures
Unused parameters and hard-coded paths reduce resilience and leave available context unused in the Codex prompt

The Bazel workflow is the more mature path and is the right foundation to extend. The proposed architecture on the next slide reflects these observations.
