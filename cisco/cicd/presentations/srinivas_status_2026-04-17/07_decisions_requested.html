Decisions Requested Today

Four directional decisions that will shape how the team sequences the next two weeks of work. Each comes with a recommendation; final direction sits with you.

1
Scope Model for Overlapping Tooling
Generalize As Separate, or Generalize In Place
When Cisco teams build tools that overlap with reusable infrastructure (Pulse and Scribbler as the live example), two patterns are available. Either extract the common capabilities into a separate, truly reusable project that serves multiple tracks, or require the standalone tool itself to be built as modular, reusable infrastructure from the start. Our recommendation is the separate project pattern because it respects the modularity directive without disrupting in-flight team ownership.
Which pattern do you want applied when team-owned tools overlap with the infrastructure we are building?
2
Error Catalog Service Scope
Scoping the Catalog Construction Effort
The Tier 1 deterministic classifier draws from an Error Catalog Service with three sources: Bazel release catalogs (version-locked, low frequency), historical NX-OS build logs (requires curation), and an approved-fix feedback loop from Tier 3 outputs. Construction effort is a sub-project, not a flag flip. Our recommendation is to scope an initial pass against Bazel catalogs plus a representative slice of recent historical logs, then grow from there.
What coverage target and timeline do you want for the initial catalog? Full historical corpus, a representative slice, or Bazel-only to start?
3
Log Ingestion Latency
CI Versus CD Pipeline Expectations
CI builds run with engineers actively waiting, so latency on that path translates directly to developer idle time. CD nightly builds run asynchronously and can tolerate batched processing with lower operational complexity. The real trade-off is engineer-wait latency against queue and resource overhead. Our recommendation is near-realtime ingestion for CI, scheduled batches every 15 to 30 minutes for CD, with a shared parse and classification pipeline.
Is near-realtime the expectation for CI, and does the batched cadence work for CD, or is there a unified target we should design to?
4
Alternative Deployment Path
Unblocking Development While Access Resolves
DeepSight access has been pending for approximately four weeks, gated on the permanent ADS machine. Even with Nexus T repository access, without a deployable environment there is no way to test or iterate. The options are continue waiting for DeepSight, stand up a local deployment path on the team's own infrastructure, or authorize deployment into an adjacent Cisco environment. Our recommendation is a local deployment path on the team's machines now, with migration to DeepSight once access clears.
Should the team proceed with a local deployment path in the meantime, wait for DeepSight, or deploy into an adjacent environment you can authorize today?

The team is ready to move on each of these as soon as direction is confirmed.
