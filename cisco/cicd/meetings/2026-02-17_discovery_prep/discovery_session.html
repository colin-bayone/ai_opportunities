04

Technical Understanding

Developer Box (Item A)

We need to understand the current state before designing instrumentation.

  1. Can we schedule a walkthrough of the local development workflow? (developer writes code, runs tests, submits PR)
  2. What testing frameworks are used locally? (pyATS, pytest, custom frameworks)
  3. How standardized is the local dev environment across teams, or does it vary significantly?
  4. What specific telemetry data would be most valuable? (test names, pass/fail, duration, coverage metrics)
  5. What is the appetite for a lightweight agent on developer machines vs. alternative collection mechanisms?

Branch Health / CD Health (Item F)

  1. What specific information do release leads need that they lack today?
  2. Are there existing dashboards or reports for release leads, or is this capability net new?
  3. What does "automated follow-up identification" look like in practice? (WebEx notifications, Jira tickets, email escalations)
  4. What criteria distinguish actionable failures from noise that shouldn't trigger follow-up?

Internal Documentation

  1. Is there existing documentation on the 39 gates - what each gate checks, expected inputs/outputs, common failure modes?
  2. Is there documentation on CDT (Context Driven Testing) architecture and how it integrates with the pipeline?
  3. Are there internal developer standards or contribution guidelines we should review?
  4. Where does this documentation live, and can we get access?
