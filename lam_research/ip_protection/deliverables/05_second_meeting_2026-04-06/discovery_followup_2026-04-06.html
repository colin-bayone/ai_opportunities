The Problem

The Core Challenge

Lam Research services competing customers simultaneously, including TSMC, Samsung, Intel, and Micron. Years of field service, troubleshooting, and escalation work have built a deep internal knowledge base. The technical knowledge within that base is largely generic and transferable across customers, but it is contaminated with customer-identifiable information: customer names, fab identifiers, site-specific details, and proprietary process references.

Because there is no reliable way to separate the generic knowledge from the customer-specific information, everything is restricted by default. The result is that Lam's own engineers cannot access solutions that already exist within the organization.

The Business Impact

Impact Area Description
Throttled productivity Support efficiency is constrained by the inability to share knowledge safely across customer boundaries. Engineers spend time re-solving problems that have already been resolved in a different customer context.
Blocked feedback loop The troubleshooting workflow (self-help, ask for help, escalate) is designed to drive resolution at the lowest tier. The feedback loop from escalation back to self-help is broken because cleaned solutions cannot re-enter the general knowledge base.
Repeated escalation costs Problems that have already been solved get re-escalated because the solutions are locked behind customer restrictions. This costs time, money, and customer trust.
Fragmented search Over-restriction has driven the creation of multiple segmented search systems, each covering a different knowledge pool with different access rules, compounding the problem rather than solving it.

Use Cases Under Discussion

Two high-level use cases have been identified, both tied to the troubleshooting workflow:

Enabling Self-Help Search

Processing stored content to remove customer-identifiable information so the underlying technical knowledge can be surfaced across customer boundaries. Users searching for solutions would see relevant results regardless of which customer engagement originally produced the answer.

Protecting Escalation Entry Points

Detecting potential customer IP at the point of data entry and notifying users before the information enters the system. Awareness and deterrence at the source, catching policy violations before they propagate into the knowledge base.

These are broad, high-level use cases. The specific applications, data types, and workflows involved have not yet been fully defined.

The Infrastructure Landscape

The technology environment is highly heterogeneous: a mix of on-premises systems, cloud-hosted systems, systems in the process of migrating to cloud, internally hosted AI tools, and cloud-based services. There is no unified data lake. Data is fragmented across many systems with different segmentation and access models. There are six or more search systems covering different knowledge pools. The organization aspires to move cloud-first and has expressed interest in a microservice architecture long-term.

To validate any approach against the selected application, we need to see examples of its actual content:

If security or IP concerns prevent sharing actual content, sanitized examples or synthetic representations that preserve the structure and patterns of real data are also valuable.

BayOne's Approach

BayOne has built detection and IP protection systems in similar environments, including semiconductor equipment companies with defense and export control constraints. The following reflects what has been proven to work, adapted to what we understand about the Lam environment so far.

This is a proven approach that will be adapted to the specific target application and infrastructure.

Hybrid Detection Architecture

Deterministic Layer

For known, enumerable entities, deterministic matching provides the fastest and most accurate detection. Zero false positives on known patterns because it is exact matching against curated dictionaries, not probabilistic AI.

  • Entity lookup tables for customer names, fab identifiers, and their variations
  • Pattern matching for structured identifiers with predictable formats
  • Maintained as configuration, not model weights

Enterprise tooling such as Microsoft Purview provides this capability with custom Sensitive Information Types designed for domain-specific entities.

AI Classification Layer

For content where deterministic matching cannot reach, AI-based classification provides contextual analysis.

  • Contextual sensitivity: determining whether a mention is sensitive in its specific context
  • Unstructured text analysis for freeform descriptions and transcripts
  • Document-level classification for overall risk assessment

Hosted on Azure AI Foundry for scalability, manageability, and integration with enterprise identity and compliance controls.

Common Data Plane

Both detection layers are unified through a common data plane hosted in Azure. This provides a single, governed layer through which all applications connect, with centralized ingestion, credential and access management integrated with IAM and RBAC, and compliance controls that IT teams can manage directly.

Content is analyzed as it enters the system. Clean content proceeds into the knowledge base immediately. Flagged content is routed for review with specifics on what was detected and where. New content entering the system is clean by construction. Historical content already in the system is processed through the same pipeline as a bounded, one-time cleanup effort, one application at a time.

Why Azure

Azure Advantage Description
Scalability Auto-scales with usage. The system grows with the organization without requiring infrastructure re-architecture.
Maintainability Accessible to IT teams to manage, monitor, and operate directly. Detection rules, entity dictionaries, and classification models are updated through standard tooling.
Compliance integration Native integration with identity and access management, role-based access control, and data governance capabilities out of the box.
Cost efficiency Enterprise Azure services like Purview and AI Foundry provide mature, supported capabilities at a fraction of the cost of building and maintaining equivalent custom infrastructure.
Production maturity Azure deployments grow into production systems that the organization owns and operates. Custom or on-premises deployments can succeed, but the maturity tends to remain at proof-of-concept level rather than scaling into operational systems.

The same architecture can work on-premises or in a hybrid model if required. Azure is not the only option, but it is definitively the most scalable, maintainable, and cost-effective path. This is the approach that worked in practice at a comparable semiconductor equipment company, and BayOne has tried the alternatives as well.

The architecture is designed so that once the first application is solved, expanding to additional applications follows the same pattern with decreasing effort. The goal is to empower the team to own and operate the capability independently.
