Technical Analysis

Why the Prior Approach Failed

The prior effort's fundamental limitation was not model selection. It was the complete absence of ground truth. The team had no golden set, no labeled training data, and no structured evaluation methodology. They worked with three reference lists (customer names, fab identifiers, and an acronym exclusion list of approximately 3,000 entries), which are configuration inputs for a deterministic system, not training data for machine learning models.

The models themselves (Hugging Face Flair, SpaCy, Azure AI) were selected through a bake-off of pre-trained models evaluated on out-of-the-box performance against a problem none of them were designed for. Presidio, the starting framework, is built for standard PII detection (social security numbers, credit cards, phone numbers). Lam's problem is custom domain-specific text classification for entities that exist only within their operational context. Applying generic PII tools to a domain-specific problem is a category error regardless of how many models are evaluated.

The parallel reconciliation approach (running all three models and comparing outputs) produced a marginal 4 percentage point improvement (21% to 17%) for the complexity of orchestrating three models. The models made correlated errors because they share the same limitation: none had any concept of what Lam's sensitive entities actually are. The 18-month timeline with no meaningful advancement reflects the gap in internal AI expertise, not problem difficulty.

Key takeaway: The prior effort was purely exploratory. Models were given raw input and evaluated on unguided output. No ground truth was established, no structured evaluation protocol existed, and the team was searching for a model that already understood their problem rather than building the detection capability themselves.

The Ground Truth Problem

The team was told that creating a golden set required a full labeling exercise estimated at over 1,000 person-hours. This estimate was accepted without challenge and became the justification for abandoning the ground truth pathway entirely. The estimate has no analytical basis. The team never scoped the actual data set, never determined what a statistically meaningful sample would look like, and never considered tiered or automated approaches to labeling.

The three-tier labeling framework presented in the meeting reframed this constraint:

Tier What It Contains Effort Level
Tier 1 Flagged entity on its own. Binary: sensitive or not. No document context. Lowest. The existing thumbs up/down data (~1,000 examples) is Tier 1.
Tier 2 Flagged entity plus the document or text field it appeared in. Context without human explanation. Moderate. Can be accelerated with auto-labeling using existing reference lists as seed data.
Tier 3 Flagged entity, plus document, plus human reasoning for why it is or is not sensitive. Highest. This is what the 1,000-hour estimate assumed for the entire data set.

The correct approach is to start at Tier 2 and subset to Tier 3 for the hardest cases. Mikhail's immediate self-assessment ("We'd be definitely Tier 1 first") showed he understood both the current position and the incremental path forward. Exploratory Data Analysis on the existing data will determine sample sufficiency before any labeling commitment is made.

Layered Architecture

The proposed approach uses a sequential funnel rather than the parallel model the prior effort employed:

Deterministic Layer

Known patterns handled through enterprise tooling (Microsoft Purview custom Sensitive Information Types). Customer names, fab identifiers, and their variations matched deterministically. Zero false positives on exact matches. Resolves the majority of cases at minimal compute cost.

AI Classification Layer

ML/NLP for pattern-adjacent cases, Gen AI for contextual ambiguity. Only processes what the deterministic layer cannot resolve. Each layer handles a smaller, harder subset. Cost and complexity scale with actual difficulty rather than total volume.

This contrasts with the prior approach of running three models in parallel and voting on results. The linear funnel is more accurate (each layer is optimized for its problem class), more cost-effective (expensive compute is only invoked when cheaper methods fail), and more debuggable (errors trace to a specific layer rather than an opaque reconciliation).

Mikhail recognized the distinction immediately: "So we just basically accidentally picked literally every single thing, one of each." This was a significant credibility milestone. He arrived at the conclusion independently after seeing the structured approach, rather than being told the prior work was wrong.

Deployment Environment

Azure is confirmed for the POC. The prior effort's Azure environment is still partially running (Azure AI infrastructure, data retrieval jobs disabled but intact), which can accelerate setup. Three deployment modes were clarified during the meeting:

Mode Description Status
Cloud (Azure) Azure AI Foundry for model hosting, Purview for deterministic detection, enterprise compliance integration Confirmed for POC. Existing environment available.
On-prem (data center) Models hosted on internal servers Not recommended. Operational burden without corresponding benefit. Compliance risk.

The on-prem Kubernetes deployment used in the prior effort created an infrastructure and maintenance burden that is unsustainable for a team without dedicated AI operations expertise. Azure AI Foundry eliminates this overhead and places the infrastructure within the enterprise compliance perimeter from the start, which also addresses the looming concern of non-compliance as the project moves from experiment to production.

Microsoft Purview as the deterministic layer serves a dual purpose: it is the right technical tool for known-pattern detection, and it bridges the organizational gap between Brad's team (CSBG) and the IT/engineering function (GIS) by placing detection capability within enterprise compliance tooling that IT already manages.

Detection vs. Redaction

Mikhail continues to frame detection and redaction as separate use cases with different technical requirements. From a technical standpoint, this is a false separation. Detection is a prerequisite step within redaction. The pipeline detects the sensitive entity, then takes an action: notify the user (detection), mask the text (redaction), block the upload, or quarantine for review. The detection engine is identical regardless of the downstream action.

Additionally, if the ingestion pipeline detects and prevents sensitive content from entering the knowledge base, the need for batch redaction of stored content approaches zero. Redaction becomes a bounded, one-time historical cleanup rather than an ongoing operational mode.

This distinction should be addressed in a separate focused conversation with Mikhail. For the POC, the team is focused on detection only, which avoids the issue for now.
