### 3. Framework Content: Eight AI Acceleration Ideas for Sephora

**Program Context:**
- Three-year EDW re-engineering program (year one in progress)
- Target completion: 2027-2028
- Approach: Re-engineering (not lift-and-shift migration)
- Source systems: SQL Server, IBM Cognos reporting, IBM DataStage ETL
- Target platform: Databricks on Microsoft Azure with retained Cognos front-end
- Philosophy: "Progress over perfection"

**Program Scale:**

| Metric | Value |
|--------|-------|
| Cognos Reports | 6,000+ |
| SSAS Cubes | 8 |
| KPIs | 800+ |
| Dimensions | 300 |
| Source Systems | 20+ |
| Timeline | 3 years (targeting 2027-2028) |
| Track execution | Finance (near completion), Merchandising/Supply Chain (next), Stores/E-commerce (following) |

**Five Key Challenges Addressed:**

1. **Capacity Constraints** — Processing 6,000 reports individually is infeasible; requires batch re-engineering approach grouping similar reports with common structures and patterns.
2. **Legacy Business Logic** — 15-20 years of business rules embedded across Cognos definitions, SSAS calculations, DataStage transformations, and stored procedures; original developers departed; outdated documentation.
3. **SME Bandwidth Constraint** — Subject matter experts responsible for validation also manage production systems; modernization cannot consume all their time.
4. **Validation Requirements** — Every re-engineering phase requires human verification (logic extraction, source-to-target mapping, transformation QA, business user sign-off); cumulative validation across thousands of assets creates throughput constraint.
5. **Timeline Constraints** — Can Finance-validated architectural patterns scale across remaining tracks without proportionally scaling team?

**Current AI Tooling Landscape:**
- Actively using: Lutra, Flow
- Under evaluation: Databricks AI capabilities
- Assessed: Various partner accelerators
- Strategy: Complement existing tools rather than create fragmentation

**Eight AI Acceleration Ideas:**

1. **Report Similarity Clustering** — Analyze Cognos report metadata (columns, joins, filters, output structures) to group reports by structural similarity. Process similar reports as batches rather than individually. Example: cluster of forty reports pulling from same three tables addressed as family with one solution pattern.
2. **Business Logic Extraction** — Parse Cognos report definitions and embedded SQL to extract calculations, filters, and business rules into structured, readable catalog. Accelerates SME review by providing readable format instead of requiring legacy code tracing.
3. **Dependency Mapping** — Trace relationships between source tables, views, ETL jobs, reports, and cubes to create complete dependency graph. Prevents surprises when unexpected dependencies surface mid-project. Enables confident sequencing decisions.
4. **Schema Mapping Validation** — Automate source-to-target column mapping with confidence scoring (name matching, data type compatibility, sample value comparison). High-confidence mappings proceed with minimal review; low-confidence flagged for SME attention.
5. **KPI Lineage Tracing** — Map 800+ KPIs back to source calculations across reports, cubes, pipelines. Identify where same KPI is calculated differently in different places. Prerequisite for clean semantic layer.
6. **Change Impact Analysis** — Simulate downstream impact of any change to source systems, tables, or transformation logic across entire reporting portfolio before deployment.
7. **Documentation Generation** — Generate documentation for undocumented stored procedures, DataStage jobs, and Cognos report logic through code analysis. Makes institutional knowledge explicit and reviewable.
8. **Consolidation Detection** — Identify reports performing nearly identical functions with minor variations. Surface consolidation opportunities reducing total asset volume requiring re-engineering.
