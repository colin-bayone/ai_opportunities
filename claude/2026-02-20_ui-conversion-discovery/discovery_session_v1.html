CONTEXT

Our Understanding

From our February 9 conversation, here's what we took away.

What we heard:

  • You have a network management product that was modernized with a new UI
  • Major customers are asking for the original UI experience back
  • The reason: their systems are integrated with it, and operators are trained on it
  • There are roughly 200 screen pages involved
  • The backend has changed, so reverting isn't straightforward
  • Traditional approach doesn't fit the timeline need
  • You're exploring whether AI-assisted development could accelerate this

What we don't know:

  • The specific product or what the UIs look like
  • Whether this is web-based, thick client, or something else
  • What frameworks are involved
  • What "integrated" means in practice for these customers
  • How the backend changes affect UI compatibility
01

Opening Questions

  1. Can you walk us through the problem from your perspective? What's driving this?
  2. Did we capture the core challenge correctly, or are we missing something?
  3. What does "integrated" mean when customers say their systems are integrated with the old UI?
02

Understanding the Product

  1. What is the product? Is it web-based, a thick client, or hybrid?
  2. What framework or technology is the old UI built on?
  3. What framework or technology is the new UI built on?
  4. Are both UIs currently accessible? Could we see them side-by-side?
03

Codebase & Architecture

  1. What languages are the old and new UIs written in?
  2. How do the UIs communicate with the backend? REST APIs, GraphQL, WebSockets?
  3. What changed in the backend during modernization? Different API contracts, data models, authentication?
  4. If you tried to point the old UI at the new backend today, what would break?
  5. Is there an API compatibility layer, or is it a clean break between old and new?
  6. How is the old UI structured? Monolith, module-based, micro-frontends?
  7. Are there shared components or a design system, or is each screen largely independent?
  8. How is state managed in the old UI? Global state, component state, external store?
  9. Are there automated tests for either UI?
04

Screen Complexity

  1. Of the ~200 screens, how would you categorize them by complexity?
  2. What makes the most complex screens complex? Real-time data, multi-step workflows, heavy interactivity?
  3. Are there screens with significant business logic in the UI layer vs. just display?
  4. Is the new UI missing functionality, or is it primarily a look/feel/workflow difference?
05

Available Resources

  1. Do you have access to the old UI source code?
  2. Do you have access to the new UI source code?
  3. Is there design documentation, wireframes, or style guides for the old UI?
  4. Is there API documentation for the new backend?
  5. Is there a dev/test environment where experimental work could be deployed?
  6. Who would be our point of contact for technical questions?
06

Identifying a POC

  1. Are there a few screens that would be representative for a proof of concept?
  2. Is there a screen that's important but self-contained?
  3. What would "success" look like for a POC conversion?
  4. What constraints exist around sharing code or accessing systems?
