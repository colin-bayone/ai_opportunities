Pain Point Analysis

The team scraped and categorized messages from the NxOS CI workflow WebEx space. The dataset spans approximately three years (April 2023 through March 2026) and contains over 4,200 actionable messages across 25 categories.

Key Findings

Two-thirds of help requests receive no response

Question/Help Request is the highest-volume category (tied with Bug/Error), yet only 34% of threads receive any reply. Engineers are asking for help and receiving silence.

Build failures take half a workday to get first acknowledgment

Bug/Error and Test Failure threads average 4 to 5 hours before a first response. At the 90th percentile, engineers wait over 10 hours.

Explicit blockers get resolved in minutes

When an issue is specifically flagged as a blocker, the average first response time drops to 12 minutes. The organization responds when urgency is communicated. The gap is in identifying what should be escalated.

Channel volume has tripled over three years

Weekly message volume has grown from 20 to 40 messages per week in 2023 to 60 to 120 in 2026. The manual triage approach is not scaling with the growing engineering organization.

Recommendation

This data suggests the highest-impact first target for AI-driven triage is the Question/Help Request category: high volume, lowest response coverage, and longest average wait. An automated system that can surface relevant documentation, link to known solutions, or route questions to the right person would address the single largest gap in the current workflow.

Two build systems are in use: Gmake (nx_main branch, legacy, being phased out) and Bazel (nx_dev branch, current focus). Both produce CI builds (developer PRs) and CD builds (official nightly). CI and CD builds have different NFS paths, different log formats, and different retention policies.

Build metadata (build number, status, timestamps) is stored in MySQL. Logs are stored separately on NFS in a flat directory structure with 40 or more individual log files per build. The 13 distinct Bazel error log types are organized per image and stage (nxos64, nxos64_msx, linecardimages, core_64_n9000, and others).

Existing Automation Review

The team reviewed the existing build repair automation pipeline (PR #642 on DCN/tools) at the code level. The pipeline consists of 7 components:

  1. analyze_and_fix_build.py serves as the entry point, parsing build_log.html for :STATUS: lines and routing to the Gmake or Bazel workflow.
  2. automated_workflow.py (Gmake) and automated_bazel_workflow.py (Bazel) provide two parallel execution paths.
  3. find_earliest_error_log.py (Gmake) and bzl_error_report.py (Bazel) handle error extraction from the respective log formats.
  4. BUILD_ERROR_ANALYSIS_NO_BUILD.md contains the Codex prompt template used for fix generation.
  5. Verification builds run with up to 3 retries per attempt.
  6. Email notification with git diff is sent upon completion.

Identified Limitations

Limitation Detail
Incomplete context extraction Error extraction captures only a subset of available context. The mtime-based file selection and backward scan that stops at ARCH lines miss relevant information.
No persistence across runs Analysis results are not preserved. Workspace files are overwritten each attempt, with no traceability from a fix back to the root cause that motivated it.
Gmake inefficiency The Gmake path rebuilds all targets on every retry. The Bazel path correctly tracks remaining failures only.
Fragile configuration Hard-coded absolute paths throughout the codebase make the pipeline fragile to infrastructure changes.
Unused parameters The error_output parameter is accepted but never passed to the Codex prompt, leaving available context on the table.

The Bazel workflow is the more mature path and serves as the better foundation for improvement. The proposed architecture (Section 03) builds on this assessment.

03

Proposed Architecture

Based on the current-state review and the technical understanding gained through hands-on sessions, the team has drafted a proposed end-to-end architecture for the build log analysis system. This is initial thinking, and several decisions remain pending.

Architecture Diagram

Block-by-Block Notes

Block 1: NFS Ingestion Decision Pending

Reads Bazel CI/CD logs from the NFS mount. Two viable modes: (a) scheduled batches every 15 to 30 minutes, which is simpler and predictable, or (b) event-driven realtime using a filesystem watcher or webhook, which offers lower latency but more moving parts. The proposal is to start with 15 to 30 minute batching to match current build cadence, then revisit once queue patterns and SLA targets are confirmed.

Block 2: Parsing and Chunking

Regex/structured parsing when log formats are uniform. NLP-based chunking where formats vary across job types. NLP summarization reduces chunk size before any Tier 2 or Tier 3 call to control token cost. This is critical given 300K to 500K line files. Final parsing strategy is pending log-format verification across CI versus CD and composite versus individual files.

Blocks 3, 4, and 5: Tiered Error Evaluation

Tier 1 (Block 3): deterministic rule-based detection using the Bazel error-code registry and regex over error descriptions. Tier 2 (Block 4): a smaller specialized ML model classifies errors that Tier 1 cannot match. Tier 3 (Block 5): LLM performs root-cause investigation and fix suggestion only for novel or compound errors. Escalation is explicit, and each tier records its confidence.

Block 5.1: Auto-Fix and PR

Consumes the root cause, fix suggestion, and context from Blocks 3, 4, and 5. The LLM produces a code patch, the system branches and commits via Git, opens a PR with root-cause summary and proposed fix, and triggers build verification. A human review gate on the PR is mandatory. PR status (merged, rejected, build pass) flows back into Block 6 for traceability.

Block 6: Structured Storage (Star Schema)

Fact table build_analysis_facts (one row per error per build) linked to dimensions dim_builds, dim_errors, dim_repos, dim_branches, and optionally dim_log_files. Persists summaries, suggested fixes, confidence scores, SHA hashes, and PR numbers. Full commit diffs and PR comments are queried live, not persisted.

Block 7: MCP Tool Interface

Exposes the schema to downstream agents and users via MCP tools: query analysis results, trace an error back to a commit or PR, fetch fix suggestions, pull build-failure summaries, and correlate with live Git/GitHub state. This is the primary consumption layer for agentic AI workflows.

Pending Decisions

1
Ingestion Mode

Batched polling versus event-driven/realtime for Block 1. The answer depends on build cadence patterns and latency requirements.

2
Log Format Verification

CI versus CD log format differences need hands-on verification before the parsing strategy in Block 2 can be finalized.

3
ML Model Training Data

Availability of labeled datasets for the Tier 2 ML model in Block 4. Without labeled data, the team would need to use pre-trained models or unsupervised approaches.

4
Fix Delivery Mechanism

PR comment, new branch/PR, or email for Block 5.1. The current approach uses email. The proposed architecture includes a PR-based approach with human review.

04

WebEx Integration

The team built and deployed two capabilities for WebEx integration: a chat scraping tool and a meeting recording extraction tool. Both are operational and have been validated against live data.

Chat Scraping

A working WebEx scraper was built and deployed into the team's own WebEx space. It captures full message history with structured fields (timestamp, author, email, bot flag, text, files, thread parent), stores to PostgreSQL, and supports per-room scoping and deduplication.

The NxOS CI workflow channel analysis presented in Section 01 was performed using this capability.

Meeting Recording Extraction

A working extraction tool was built that retrieves transcripts, summaries, action items, and audio from WebEx meetings via the API.

Key constraint discovered: The WebEx API restricts recording and transcript access to meeting owners. A regular attendee cannot extract these artifacts via API.

For meetings where the team is the organizer, the API path works as expected. For broader automation across the organization, either admin-scoped tokens or an alternative approach (such as audio-file-based transcription) would be needed. This constraint affects the automation architecture for meeting recording extraction at scale.

Open Questions for Discussion

The following questions require direction before the team can proceed on several fronts. Each is structured with context to support a focused discussion.

1. Scope Clarification for WebEx Work

The team has built both a chat scraper and a meeting recording extractor. Should the focus remain on the NxOS CI workflow channel specifically, or should this be generalized for the broader set of WebEx spaces?

2. Relationship to Existing Work

The team observed the Nexus T agent in the NxOS CI workflow space. How does the work being done here relate to that effort? Should the team be collaborating with that group, building on their work, or focusing on a different aspect of the problem?

3. Fix Delivery Preference

For the build log analysis pipeline, when AI generates a fix: should the system comment on the existing PR, open a new PR with the fix, or notify via another channel? The current workflow uses email. The proposed architecture includes a PR-based approach with human review.

4. Tiered Processing and ML Model Data

The proposed architecture includes a Tier 2 ML model for error classification. Does the team have access to labeled datasets of past build errors and their resolutions? If not, should Tier 2 initially use pre-trained models or unsupervised approaches?

5. Batch Versus Realtime for Log Ingestion

For CI builds (where developers are waiting), is near-realtime processing expected? For CD nightly builds, is batch processing on a 15 to 30 minute cycle acceptable?
